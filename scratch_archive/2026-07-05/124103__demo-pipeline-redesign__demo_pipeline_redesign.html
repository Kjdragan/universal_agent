Autonomous Demo Pipeline

Current state, the real limits, and a curation-first redesign

Code-verified against deployed prod (/opt/universal_agent) · 2026-07-05

The one-line answer: the limit is 10 auto-built demos/day (a self-imposed volume guard, not a paid rate limit — they build on the flat-rate ZAI/GLM plan). You get one email per finished demo, which is the phone flood. Meanwhile ~350 candidates sit unseen in a pending-approval graveyard, ranking is by recency not quality, and no demo ever becomes a reusable skill. Your instinct to gate + curate is right — and it's half-built already.

3Three things worth correcting

1. No paid rate limit is being burned. Autonomous demos build on the flat-rate ZAI/GLM plan, not Anthropic Max. So "spending the whole morning's rate limit" isn't what's happening — the 10/day is a self-imposed volume guard. The real problem is volume + weak curation + phone noise, which is 100% valid.
2. Curation is basically absent. A binary yes/no judge + a recency sort decides what gets built. There's no notion of "which of these is most interesting / useful / worth learning." A real 0–100 value score is computed and then discarded.
3. Demos don't compound. Each one is a one-off app. The north star — demo → Agent SDK script → reusable Claude Code skill — never happens in the autonomous lane. That extraction step exists only when you run /demo by hand.

5Recommended shape

Curate → gate → digest → approve → build → distill
  • Score, don't just judge. Turn the already-computed 0–100 value score back on for the YouTube lane so candidates rank by worth, not upload date.
  • Lower the auto-gate. Drop the ceiling so few/none auto-build; the rest wait for you.
  • One ranked digest email per block with my recommendation (which to build, which to skip, why) and tap-to-approve links — replacing ~7–11 per-demo pings with one curated slate.
  • Approve a subset from the email; approved demos build (uncapped) on ZAI/GLM as they do now.
  • Distill the winners. A passing demo runs skill-creator and promotes a reusable skill into dragan-plugins — so the pipeline finally compounds into your capability library.

Roughly: one net-new digest/email primitive, one score-persistence fix, one config change, and (optionally) porting the skill-extraction step from the interactive factory. Most of it is wiring existing parts, not new machinery.
