Convergence Skill Suite — handoff for the generator/evaluator-loop session

Why you're getting this: you're building a generator→evaluator→improver loop for non-deterministic tasks (output you can't grade with a simple assert). A sibling session just built and shipped a small, verified suite that solves exactly the evaluator + accept/reject part of that loop, plus the orchestration around it. This document is so you can decide whether to reuse the code, steal the design, or ignore it. Nothing here is load-bearing for your work unless you choose it.

The single most relevant piece for you is judge-panel — a variance-aware LLM-as-judge evaluator. If you read only one section, read that one.

TL;DR

  • 5 primitives + 2 masters, each a standalone ~/lrepos/demo-* repo, all promoted into the dragan Claude Code plugin as /dragan:<name> commands.
  • The engines are plain Python, dependency-injected — you can drop a single file into your project and inject your own model/checker. The judge engine is pure stdlib; only the bundled real-LLM judge needs the anthropic SDK.
  • For non-deterministic eval, the design answer is: pairwise (not absolute) judging + a panel of N judges + a noise-floor margin so a noisy judge becomes a comparator your loop can actually gate on (keep/revert) without drifting inside the judge's own jitter.
  • Status: fully built, verified, shipped. Each repo has a deterministic offline self-check (*_OK) and, where relevant, a real ZAI/GLM path. Desktop interactive tooling — it does not deploy to the VPS. Commands go live on next session.

Mental model: primitives vs masters

Call a primitive when you already know the shape of the evaluation. Call a master when you don't, or when the work is a multi-step pipeline.

Skill The loop When
primitive autoresearch-loop scalar hill-climb; keep-if-better / revert you have a single number to optimize
primitive pass-fail-repair run a real check, repair on real failure you have a runnable pass/fail check
primitive judge-panel N independent LLM judges, pairwise, noise-floor margin output is subjective / non-deterministic — no number, no check
primitive optimize-then-verify optimize the soft objective, then verify the hard one explicit "optimize X then check it still passes Y"
primitive subagent-fanout parallel fan-out → fan-in synthesis breadth: N parallel attempts, merge
master converge routes the gate (check/score/judge/hybrid), gate-first, escalates repair→optimize→fanout one artifact to a bar, path unclear or multi-dimensional
master converge-flow a sequence of gated checkpoints + a final gate a multi-phase pipeline (each step must pass before the next)

Your generator/evaluator loop maps cleanly onto this: your generator is the propose/improve function; judge-panel is your evaluator; converge is the keep/revert+escalation harness if you want it prebuilt.

judge-panel — the part you actually want

Location: ~/lrepos/demo-judge-panel/ · GitHub Kjdragan/demo-judge-panel Engine: judge_panel.py (pure stdlib) · judges: judges.py Command: /dragan:judge-panel · Self-check: uv run python main.py → JUDGE_PANEL_OK

The core idea

A single LLM judge is noisy: it scores the same output differently across runs and flatters easily. A panel — several independent judges, aggregated with a noise-floor margin — turns that noise into a stable, gateable signal.

Two modes (prefer the first for a loop):

  • pairwise — "is the candidate better than the baseline/incumbent?" Each judge picks A or B. The panel is decisive only when the winning side clears a supermajority of all valid votes. Ties dilute (a judge that can't tell counts against decisiveness). A 3–2 split is within the noise → decisive=False on purpose. This is what stops a subjective hill-climb drifting.
  • absolute — "score this 0–10 against a rubric." Panel reports the median, decisive only when the judges' spread (stdev) is within max_spread.

Verdict.accept_candidate() is the one call your loop needs: it returns true only when the panel decisively preferred the candidate. That's your keep/revert gate.

The judge contract (this is the reuse seam)

A judge is any callable: judge(candidate, baseline, rubric, mode) -> Vote. The engine never calls a model itself — you inject the judge. So you can drop judge_panel.py into your repo and wire your own model with a ~15-line adapter, regardless of provider. The bundled LLMJudge(lens, model="haiku") is just one implementation (it routes through the operator's ZAI/GLM via lab_common); the HeuristicJudge is a deterministic one used for reproducible tests.

Signatures

from judge_panel import panel_pairwise, panel_absolute, Verdict, Vote

# pairwise: is `candidate` better than `baseline`?
v = panel_pairwise(candidate, baseline, judges, *, rubric="",
                   supermajority=2/3, max_workers=8)  # -> Verdict
v.accept_candidate()   # True iff decisive AND candidate won  <-- the keep signal
v.candidate_better     # bool | None (None = tie/undecided)
v.agreement            # winning-side fraction of all valid votes
v.line()               # human-readable one-liner

# absolute: score against a rubric
v = panel_absolute(candidate, judges, *, rubric="", max_spread=1.5)
v.score                # median across judges
v.spread               # population stdev = the noise estimate
v.decisive             # spread within max_spread

Minimal usage with your own model

from judge_panel import panel_pairwise, Vote, A_WINS, B_WINS, TIE

def my_judge(lens):
    def judge(candidate, baseline, rubric, mode):
        # call YOUR model; return a Vote
        ans = my_llm(f"Through the lens of {lens}, which is better on:\n{rubric}\n"
                     f"A:\n{candidate}\nB:\n{baseline}\nReply A, B, or tie.")
        choice = {"A": A_WINS, "B": B_WINS}.get(ans.strip()[:1].upper(), TIE)
        return Vote(judge=f"llm:{lens}", choice=choice)
    judge.__name__ = f"judge_{lens}"
    return judge

panel = [my_judge(l) for l in ("clarity", "correctness", "usefulness")]
v = panel_pairwise(new_output, current_best, panel, rubric="task spec here")
if v.accept_candidate():
    current_best = new_output          # variance-aware keep/revert

Design points worth stealing even if you don't use the code

  • Pairwise beats absolute for LLM judges. "Is B better than A?" is far more reliable than "score B 0–10". It also maps straight onto keep/revert.
  • A margin, not a bare majority. Accepting on 3–2 is accepting inside the noise. Require a supermajority (tunable) so an accepted change provably cleared the panel's disagreement floor.
  • Diverse lenses, not N identical judges. Give each judge a different perspective (clarity / correctness / usefulness); the spread becomes a free noise estimate and diversity catches failure modes redundancy misses.
  • One bad judge can't sink the panel — a judge that errors is captured and excluded; the rest still decide.
  • Anti-overfit: rotate the judge framing between proposing and accepting (a train/val split for subjectivity); keep any hard deterministic constraints on a separate runnable gate (see pass-fail-repair); a panel gives you "judged better," never "proven better."

converge — prebuilt keep/revert + escalation harness (optional)

If you'd rather not hand-roll the loop, converge is the single-artifact master.

Location: ~/lrepos/demo-converge/ · converge.py · /dragan:converge Self-check: uv run python main.py → CONVERGE_OK

It does three things your loop probably needs:

  1. Routes the gate. A Gate carries whatever you can measure — a check (runnable), a score (number), or judges (a judge-panel), or a hybrid (hard check + soft score). The kind is inferred; you don't pre-pick a loop.
  2. Gate-first. Evaluates the seed before spending anything — don't optimize what already passes.
  3. Escalates in cost order, re-gating after each tier, stopping at the first pass: Tier 0 repair → Tier 1 hill-climb-then-verify → Tier 2 fan-out.
from converge import Gate, converge
from judges import default_heuristic_panel        # or your own injected judges

gate = Gate(judges=my_panel, rubric="task spec")   # subjective gate
r = converge(seed, gate,
             propose=my_generator,                 # artifact -> improved artifact
             baseline=reference_output,            # judge gate compares against this
             ledger_path="converge.tsv")
r.converged, r.via, r.artifact                     # via: passed-first-try|repaired|optimized|fanout

For a multi-step process where each phase must pass before the next, use converge_flow (converge_flow.py, /dragan:converge-flow, CONVERGE_FLOW_OK): a list of Step(name, gate, improve/propose, ...) + an optional final_gate; the improved artifact chains forward; stop_on_fail halts at the first checkpoint that can't converge. converge-flow is manual-invocation by design.

Three ways to consume it

  1. Conversationally — in a Claude Code session with the dragan plugin loaded, invoke /dragan:judge-panel (or converge) and follow the SKILL.md.
  2. Code-level reuse (recommended for your loop) — copy judge_panel.py (pure stdlib) into your repo, inject your own judge via the judge(candidate, baseline, rubric, mode) -> Vote contract, and gate on Verdict.accept_candidate(). Add converge.py if you want the gate-first + escalation harness. No dependency on the rest of the suite.
  3. Design-only — read this doc + the SKILL.md files and reimplement the pairwise + noise-floor-margin pattern in your own stack. The idea is the value; the code is ~150 lines.

How this plugs into a generator/evaluator loop

generate/propose ─▶ candidate ─▶ judge-panel(candidate vs current_best)
                                        │
              accept_candidate()  ──────┤ decisive & better ─▶ current_best = candidate
                                        └ within-noise / worse ─▶ revert, propose again
  • Your generator = propose(artifact) -> artifact' (or improve(artifact, feedback)).
  • Your evaluator = a judge-panel over your own model judges.
  • Your accept rule = panel_pairwise(candidate, current_best, panel).accept_candidate().
  • Want gate-first + automatic escalation (repair → optimize → fan-out) for free? Wrap it in converge with Gate(judges=panel, rubric=...).

Open question for your design (worth deciding explicitly): what's your baseline for the pairwise comparison? Comparing each candidate to the current best (incumbent) gives a clean monotone hill-climb; comparing to a fixed reference gives an absolute bar. converge uses incumbent-comparison inside its hill-climb tier and reference-comparison at the gate — you may want both.
