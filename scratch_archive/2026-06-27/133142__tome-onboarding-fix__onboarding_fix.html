Tome family onboarding — diagnosis & fix

Why the snafus happened, and the unified flow that removes them.

Root cause: two systems, glued by hand

Onboarding one person actually crosses two independent systems that have no knowledge of each other. The app automates only its own half; the entire Tailscale half is manual admin-API work I did by hand this session.
Tailscale — 100% manual
  • invite to tailnet
  • accept + install + sign in
  • admin-approve (the "needs-approval" surprise)
  • ACL grant scoping them to Tome
Tome account — automated
  • mint_invites → one-time token
  • dispatch_invites → emailed link
  • set password in the web app

The repo's only link between them is the email text I added today. Nothing checks that someone is even on the network before they're handed a Tome link.

Every snafu → the same gap

The fix: one onboard command + a status board

A thin Tailscale admin client in the repo + one CLI that drives both systems from the existing family roster. No new database — it reads live state from Tailscale + the Tome store.

BEFORE — ~8 hand-coordinated steps

  • manual TS invite (admin console)
  • manual edit ACL grant
  • manual approve in console
  • manual copy their join link
  • cmd mint_invites
  • cmd dispatch_invites --tailscale-url …
  • manual diagnose if they're stuck

AFTER — one command each

  • ✦ onboard add marie
    invites to tailnet · upserts ACL scope · mints Tome token · sends ONE combined email with both links
  • ✦ onboard status — the board below
  • ✦ onboard approve marie (or --all)
  • ✦ onboard resend marie
