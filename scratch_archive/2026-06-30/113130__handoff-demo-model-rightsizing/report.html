HANDOFF — Model right-sizing of the demo_factory pipeline

For: a fresh build session (Claude Code in ~/lrepos/demo_factory). Goal: use the cheapest model that holds quality at each stage of the demo_factory pipeline — trim Opus (which bills the Claude-Code/Anthropic plan) where Sonnet or Haiku suffices, and tune the Gemini judge tiers — without losing demo quality. This is a config + empirical-analysis task, not a rebuild: the per-stage model seams already exist.

Ironic but apt: this is the same "model right-sizing" methodology as the Arize-Phoenix demo that kicked off this whole workstream — now applied to our own pipeline.

Why now

Proactive demos now run on the demo_factory /demo engine (consolidation shipped 2026-06-30, live). Every proactive build is an Opus /goal session → it bills the plan, capped at 3/day. Opus is the dominant per-demo cost. The seams to dial each stage's model are already in place; what's missing is the decision about which model each stage actually needs.

Current state — the seams (verified, with file:symbol)

Two model families are in play.

1. The BUILD — Claude / Anthropic-Max (the expensive lever)

  • scripts/_launch.py → OPUS_MODEL = "claude-opus-4-8"; goal_argv(condition, model=OPUS_MODEL) builds the claude --print … --model <model> … /goal … argv.
  • scripts/goal_build.py → --model flag (default _launch.OPUS_MODEL), threaded into goal_argv. This is the one Claude/Opus knob.
  • scripts/build_demo.py → --build-model (default claude-opus-4-8), passed to goal_build --model.
  • Hard constraint (do not violate): per CLAUDE.md ("The Anthropic-Max launch profile"), the /goal loop MUST run on real Anthropic Max so its internal evaluator is real Haiku — not the ZAI glm-4.5-air proxy. So the build model must be a real Anthropic id (opus/sonnet/haiku), and _launch scrubs all ANTHROPIC_*/forwards the OAuth token. Do not route the build to ZAI.

2. The JUDGES / ENRICHMENT — Gemini (already cheap)

  • scripts/evaluate_artifact.py → --model (default gemini-3-flash-preview) — the fidelity gate (DEMO_EVAL), majority-voted over --samples.
  • scripts/land_demo.py → --eval-model (default gemini-3-flash-preview) → evaluate_artifact --model.
  • scripts/build_demo.py → --eval-model (default gemini-3-flash-preview).
  • scripts/validate_skill.py → --eval-model; scripts/make_video_brief.py → --model; scripts/publish_demo_exhibit.py → --model (all gemini-3-flash-preview).

What does NOT exist

  • No central model config, no per-stage/per-demo-class tiering, no right-sizing analysis. STATUS.md/PLAN.md don't mention it. The deterministic verifier (verify_demo.py) is correctly model-free — leave it.

Verify current model ids from official docs before coding (per repo CLAUDE.md "Research authority"): https://code.claude.com/docs/ (Claude ids) — a deprecated id = a failed build. Sonnet/Haiku ids change; don't trust memory.

The work

  1. Define demo classes (model need differs by class). Suggest using source_detect.py's archetype (agent_sdk vs generic) plus a complexity signal: e.g. simple-local-compute (Mandelbrot-class, no API), framework/API demo, agent-SDK skill, visual/multimodal.
  2. Run the build at each Claude tier (goal_build --model claude-opus-4-8 | claude-sonnet-<id> | claude-haiku-<id>) for a representative seed per class. Capture: did it pass DEMO_VERIFY + DEMO_EVAL, how many /goal rounds, wall-time.
  3. Score quality per tier using the existing gates as the metric — verify_demo pass, evaluate_artifact fidelity (gating + holistic), build success-rate/rounds — same shape as the Phoenix right-sizing demo.
  4. Pick the cheapest tier per class that holds ≥ a threshold of the Opus baseline (e.g. ≥95% fidelity AND equal verify pass-rate). Right-sizing is not only down: if a heavier Gemini judge measurably catches more fidelity misses on visual/multimodal demos, size up there.
  5. Encode the decision as a small config (e.g. scripts/model_tiers.py — a {class: {build_model, eval_model, eval_samples}} map, with an env override) and wire it through the existing --model/--build-model/--eval-model seams. build_demo.py selects --build-model from the class (derived from source_detect); the proactive objective (universal_agent/services/cody_dispatch.py) can pass the class or let build_demo infer it.
  6. Default the proactive path to the right-sized config; keep Opus as the high tier for the hard classes. Make all-Opus a one-flag fallback.

Acceptance criteria

  • For each demo class, the chosen tier holds verify pass-rate == Opus and fidelity ≥ threshold of the Opus baseline, on the validation set.
  • Measurable Opus-token reduction on the proactive path (the cost goal).
  • The model choice is a single config lever; reverting to all-Opus is trivial (env or one constant).
  • No ZAI on the build path; verifier stays model-free; capability-honesty preserved (don't let a cheaper model squeak past a gate it shouldn't).
  • A short evidence report (quality vs tier per class) accompanies the config.

Files likely touched

  • scripts/_launch.py, scripts/goal_build.py, scripts/build_demo.py — build-model selection + read the config.
  • New scripts/model_tiers.py (or a [tool.demo_factory.models] block in pyproject.toml) — the class→model map + env override.
  • scripts/evaluate_artifact.py / land_demo.py / validate_skill.py --eval-model — Gemini tier per stage.
  • scripts/selftest.py — add a check that the config resolves a model per class and that build_demo threads it.
  • (universal_agent) services/cody_dispatch.py proactive objective — optionally pass the demo class.

How to run the analysis without burning the plan

  • Baseline once on Opus per class; then Sonnet/Haiku (cheaper) for the comparison runs. Bound the sample (2–3 seeds/class). Use --build-only to skip land/promote side effects during analysis.
  • Cheap exerciser seed: "render a Mandelbrot set PNG with numpy and pillow" (--demo-id …), already validated end-to-end on the VPS.

Context / provenance

  • The build is Opus today by default only (_launch.OPUS_MODEL); the /goal internal evaluator is already real Haiku — so the agent model is the lever, not the eval loop.
