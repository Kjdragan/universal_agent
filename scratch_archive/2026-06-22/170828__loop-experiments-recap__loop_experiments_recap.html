loop_experiments — the loop-it lab

A skill that turns a request into a bounded, self-terminating agent loop — and four live loops that prove it.

An agent loop is a task with a check: do one bounded action, run the fixed check, record evidence, then continue or stop. loop-it is the thin glue that selects an in-house loop pattern — or designs one from scratch when none fits — composes a structured prompt, and runs it. Fully self-contained: no external library, no URL.

What loop-it is made of (all in-house)

PartRole
loop_patterns/GRAMMAR.mdthe runtime source. The loop grammar (goal / bounded action / fixed check / record / stop) + safety rails + the non-deterministic-check discipline. The load-bearing value, owned outright.
loop_patterns/PATTERNS.md~5 curated worked-example shapes. Selection rates the fit — strong (use as-is) / adapt (use + state the change) / no-fit (design from scratch). Not a menu you must order from.
native /loopthe execution primitive for the durable engine: self-paced via ScheduleWakeup.

loop_kit (the deterministic spine)

PieceJob
state.pybudget guard: start / record / status. status exits 3 once the check passes or the iteration cap or the wall-clock budget is hit. The guard — not the agent's opinion — stops the loop.
judge.pyan LLM-judged check: an independent GLM model scores an artifact against a rubric, run N times, majority verdict wins.
catalog.py + loop_library/a frozen 50-loop reference snapshot — an offline idea bank for designing new patterns. Not the runtime source, never re-fetched, no URL.

Two engines

fast

Iterate back-to-back in one session, no waits. Best when the check runs in seconds and the task converges in a few steps. (small / medium / stretch demos)

durable

Native /loop self-paced: each iteration schedules the next via ScheduleWakeup (60s–1hr apart), state persists on disk, ends by not rescheduling once the check passes. Survives an idle session. (durable demo)

The four loops — evidence

1 · small_calc fast converged

Loop #027 autonomy-loop builder-reviewer (adapted) — implement code until a frozen pytest suite goes green. The pytest suite is the gate.

iterbounded changecheck
0stub (bare number)7 failed, 3 passed
1tokenizer + additive (+ -)6 failed, 4 passed
2precedence (* /), unary, ÷-by-zero2 failed, 8 passed
3parentheses10 passed, 0 failed → STOP

2 · medium_coverage fast converged

Loop #005 the 100% test coverage loop — here the loop adds tests to close a frozen module's branches (the mirror image of #1, which added code). Gate: --cov-fail-under=100.

itertests addedbranch coverage
0thin happy-path suite79%
1to_roman error branches87%
2from_roman errors + round-trip100% → STOP

3 · stretch_pitch fast converged · non-deterministic check

Loop #034 multi-LLM convergence — author = Claude, judge = an independent GLM. The check is a judgment, not an exit code. Rewrite a weak pitch until the judge approves it on a rubric.

stepartifactjudge (3 votes)gate
0buzzword soupfail/fail/fail · totals 6/4/5majority fail
1concrete rewrite (named user, problem, mechanism)pass/pass/pass · 20/20/20majority pass
1c(same text, independent round)pass/pass/passstreak confirmed → STOP

How a fuzzy check is made usable: (1) majority-of-N absorbs per-run variance, (2) a confirmation streak before declaring done, (3) a hard cap that escalates instead of spinning.

Honest finding. 6/4/5 → 20/20/20 in one step is too clean — glm-5-turbo rubber-stamped a perfect score the moment the buzzwords left. A weak judge passes the gate too early. Non-deterministic loops work only if you vote, require a streak, and keep a hard cap — and the judge's quality, not the loop, becomes the thing to invest in (stronger model, anti-gaming rubric, held-out cases).

4 · durable_build durable converged across a real wakeup

Loop #030 five-minute repository maintainer (closest fit) — assemble a file one section per wakeup, proving the native /loop self-paced mechanism.

iterwhenchangecheckdecision
1turn Nwrote ## Fast engine1/2fail → ScheduleWakeup(60s)
2turn N+1 (wakeup)wrote ## Durable engine2/2pass → STOP, no reschedule

Termination is the absence of the next ScheduleWakeup — bounded by the same state.py guard as the fast engine. One stop policy, two engines.
