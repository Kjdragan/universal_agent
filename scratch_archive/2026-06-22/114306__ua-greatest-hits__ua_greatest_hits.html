UA is a set of long-lived Python services that run only on the VPS (uaonvps, user ua, /opt/universal_agent). The desktop is Kevin's interactive cockpit — he develops there via Claude Code, but no service, timer, or worker ever runs on it. Since the autonomous-runtime split (LIVE 2026-06-19), production runs two gateway_server processes: the public HTTP gateway on :8002 owns WS/dashboard/exec traffic, and a second process (role=autonomous_worker, :8092) hosts every autonomous loop off the HTTP loop. loop_control.py::should_host_autonomous_runtime makes them mutually exclusive — exactly one process runs the loops, so no double-fire.

  • Why split: glm-5.2 thinking-ON turns ran 15+ min in-process and starved uvicorn ("Gateway unreachable"); moving them off the HTTP loop is the structural fix.
  • Principals: Simone (always-on heartbeat orchestrator) routes work to on-demand VP workers — CODIE (coding/PRs), ATLAS (general), and opt-in HOMER (ATLAS capacity twin, peak ≤ 2 running).
  • All work normalizes into the shared activity_state.db Task Hub; VP missions live in vp_state.db; intel in csi.db.

UA is decomposed into independent systemd units restarted together by the deploy workflow.   • Don't confuse :8080 with UA — it's filebrowser, a co-tenant on the box. Treating it as a UA gateway port has misled prior diagnoses.
  • To re-verify the deployed code: curl -s localhost:8002/api/v1/version should equal origin/main; systemctl list-units 'universal-agent-*' enumerates the live fleet.

Two schedulers (and the gotcha that fools everyone)

Recurring work runs on two independent substrates — deploy-independent systemd timers and the in-process CronService — and a migrated job that reads enabled:false is correctly migrated, not off.

(global rollback)"] -.-> A classDef live fill:#e7f5e7,stroke:#2e7d32; class T1,T2,C1,C2,C3,C4 live;

UA splits recurring work across two schedulers. Substrate (A) is roughly twenty systemd timers — the migrated set enumerated in systemd_migrated_jobs.py::SYSTEMD_MIGRATED_SYSTEM_JOBS — each a deploy-independent OnCalendar unit that fires regardless of gateway restarts. Substrate (B) is the in-process CronService running inside the autonomous worker, where only four jobs are genuinely live: simone_chat_auto_complete (*/1), vp_mission_pr_reconciler (*/15), paper_to_podcast_daily, and morning_ideation_report — these stay in-process because they need the agent runtime, skills, or MCP.

  • The gotcha that fools everyone: a migrated job still appears in cron_jobs.json marked enabled:false. That is the correct migrated state — the timer is the sole firer — not "this job is off." Reading the flag literally is the #1 false "is it dead?" alarm.
  • Disambiguate by name with the predicate systemd_migrated_jobs.py::is_migrated_to_systemd — never trust the raw enabled field for a migrated job.
  • Global rollback: set UA_SYSTEMD_TIMER_MIGRATION_DISABLED=1 to fall the whole migrated set back to the in-process scheduler.
  • The remaining cron_jobs.json entries (28 total) beyond the four live ones are migration artifacts, not active in-process jobs.

Simone's heartbeat loop drives proactive findings in the autonomous worker, while a single reconciled activity inventory leads the 3×-daily report and answers "is the whole fleet healthy right now?"

Simone's proactive brain is heartbeat_service.py::HeartbeatService — an async loop hosted in the autonomous worker. Each tick it runs an LLM turn driven by memory/HEARTBEAT.md, advances proactive mission work, and emits findings. Those findings flow to email and Mission Control — the heartbeat deliberately does not write Task Hub rows (that path was decoupled). Alongside it, services/proactive_activity_report.py::build_activity_inventory is the fleet-liveness surface: it reconciles three planes — systemd timers, in-app crons, and lane DB-freshness — into a single N healthy · N degraded · N paused · N dark line that leads the 3×-daily proactive report. It is pure-Python, deterministic, and never raises. Think of it as the runtime equivalent of the Platform Status Registry: the registry is the dated doc snapshot; this inventory is the live answer.

  • healthy ✅ — running on cadence
  • degraded ⚠️ — running but stale / behind expected cadence
  • paused ⏸️ — deliberately turned off by the operator
  • parked ⏸️ — intentionally not running (e.g. hackernews)
  • dark 🌑 — never ran and not parked
  • unknown ❔ — referenced but unverifiable

Five physically separate SQLite files with deliberate lane isolation — and a couple of stale orphans that trick diagnostics.

vp_state.dbLive VP mission ledger for external workers — vp_sessions, vp_missions, vp_events, bridge cursors (CODIE/ATLAS/HOMER).Resolve via get_vp_db_path(), NOT runtime_state.db. Cross-VP concurrency must read live mission state here, not the dispatcher's assignments view.

runtime_state.dbSimone's in-process control plane — run queue, attempts, steps, tool-call ledger, checkpoints, leases.It is the live run queue, not a cache — deleting it drops queued/running runs. The Task Hub backlog health check wrongly queries here, so it reports ~0 backlog.
.csi_digests.dbLean CSI digests (the link-first daily/hourly digest store).Separate from csi.db; don't conflate digest rows with the raw event/analysis stream.

Segregation is deliberate: low-priority background writes (CSI, Task Hub) must never contend on a write lock with high-priority agent-session writes. Every runtime/activity/VP connection opens through connect_runtime_db with WAL + autocommit (isolation_level=None) so the write lock is held only per-statement. The single most misremembered fact: the Task Hub schema lives inside activity_state.db — a 2026-06-05 incident forked 18 undelivered intel briefs into a cwd-relative orphan because a skill used a placeholder path instead of get_activity_db_path().

  • WAL sidecars (-wal, -shm) travel with each DB — never bare-copy just the .db; use the SQLite backup API.
  • Schema is created lazily on first open via ensure_schema() — there is no standalone migrate step.
  • When a number looks wrong, confirm which DB it came from first — orphans and stale relics diverge wildly in row counts.

The Virtual Principals (VPs): CODIE, ATLAS, HOMER

A VP mission is the unit of delegated autonomous work, executed by a long-running worker that runs in production as a systemd template instance universal-agent-vp-worker@<id>.

VPidinferencerole / outputprod status
CODIE (a.k.a. "Cody")vp.coder.primaryZAICoder — builds runnable demos / code; output is a PR; runs cli mode; success = completed_with_prLIVE (default on)
ATLASvp.general.primaryZAIGeneralist — research, intel-brief synthesis, convergence evaluation; runs sdk mode in-processLIVE (default on)
HOMERvp.general.secondaryZAIATLAS capacity twin — shares ATLAS_SOUL.md; takes general-pool overflow only when ATLAS is busy and CODIE is idleLIVE in prod (code default OFF; enabled via UA_VP_ENABLED_IDS)

The cardinal rule is that inference follows the VP, not the work: each VP profile carries its own inference_mode (all three default to ZAI/GLM), so a research mission dispatched to CODIE still runs on CODIE's backend and a coding mission sent to ATLAS runs on ATLAS's — route work by VP, or set a per-task cody_mode override. Concurrency is deliberately capped: peak concurrent running VP missions stays at ≤ 2. ATLAS holds its own slot, and the second slot is shared between CODIE and HOMER under a bidirectional mutual exclusion — CODIE defers when a HOMER mission is running, and HOMER only spills when no CODIE mission is running. Caps: UA_MAX_CONCURRENT_VP_CODER=1, UA_MAX_CONCURRENT_VP_GENERAL=2. The gate keys on live vp_missions RUNNING state in vp_state.db, not the assignment snapshot (which is blind to running VP missions).

  • HOMER is a capacity twin, not a new persona — nothing ever tags preferred_vp=vp.general.secondary, so it only ever takes pool overflow.
  • A completed_without_pr result from CODIE is a likely-failure signal — Simone's digest distinguishes a shipped PR from a no-op.
  • Workers survive deploys: deploys no longer restart universal-agent-vp-worker@* units; code currency comes from a between-missions self-restart.

Which model runs where (the 3 execution profiles)

UA runs Claude under three distinct execution profiles; confusing them is the single most common source of model-routing mistakes, because the profile decides the endpoint and credential — not the model string.

ProfileWho / whenModel
1. Interactive codingKevin at a terminal (claude / claudereal / Antigravity)Real Anthropic Max via OAuth (true Opus / Sonnet / Haiku); ANTHROPIC_* stripped so OAuth wins
2. UA autonomous in-processSimone heartbeats, dispatch sweep, ATLAS, intel cronsZAI / GLM (cheap); ZAI-routing vars injected at service start by initialize_runtime_secrets()
3. Cody / all-VP per-task subprocessVP-dispatched coding / demo missionsZAI by default (re-flipped 2026-06-07); Anthropic Max is opt-in per-task via cody_mode="anthropic"

These two axes are orthogonal: the tier resolvers in utils/model_resolution.py always return a model name, while the profile decides what that name means. resolve_opus() → glm-5.2 (the opus tier since 2026-06-13; thinking defaults ON and is 10–24× the tokens, so cheap crons pass thinking={"type":"disabled"}); resolve_sonnet() → glm-5-turbo; resolve_haiku() → glm-4.5-air (operator-locked). The single most important fact is that routing is decided by ANTHROPIC_BASE_URL, not by the model string — a bare Anthropic() in any daemon process hits ZAI because the ZAI base URL + key sit on os.environ; reaching real Anthropic means removing those vars so the CLI falls back to api.anthropic.com + OAuth, which is exactly what the interactive launcher and the Cody anthropic-mode env-scrub do.

  • Common past mistake: "Cody defaults to Anthropic Max" is WRONG since 2026-06-07 — Anthropic began API-billing the Claude-Code-via-Max SDK path, so Cody and all VPs default to ZAI. Anthropic Max is the opt-in (per-task / per-VP pin / dashboard tile / UA_CODY_DEFAULT_MODE).
  • When a mission resolves to cody_mode="anthropic", execution_mode="cli" is forced and every ANTHROPIC_* var is scrubbed from the subprocess so OAuth engages.

Ship → verify (the deploy pipeline)

Any branch becomes running production through one GitHub-Actions path: PR → CI gate → squash-merge → VPS deploy → live-SHA confirmation.

sequenceDiagram actor Dev as Dev / Agent participant PR as GitHub PR to main participant Val as pr-validate.yml participant AM as pr-auto-merge.yml participant Dep as deploy.yml participant VPS as VPS remote_deploy.sh participant Ver as api/v1/version Dev->>PR: branch off fresh origin/main, open PR PR->>Val: pull_request to main Val->>Val: py_compile + ruff + pytest tests/unit + tripwire alt gates fail Val-->>PR: ci-failure issue filed else gates pass and non-draft AM->>PR: auto-merge --squash (AUTO_MERGE_PAT; excludes codie/kevin/feature) PR->>Dep: push to main fires Deploy (concurrency guard, no cancel) Dep->>VPS: SSH-pipe remote_deploy.sh VPS->>VPS: git reset --hard origin/main, reinstall units, restart VPS->>VPS: parallel health gate (gateway, api, webui) Dep->>Ver: deploy-notify curls live version Ver-->>Dev: Telegram OK when live SHA equals origin/main (Rule A) end

The model is uniform: branch → PR → CI gate → squash-merge to main → the main push triggers deploy.yml, which SSH-pipes remote_deploy.sh to the VPS, fast-forwards the checkout, reinstalls systemd units, restarts services, and health-gates → a separate notifier confirms the live SHA. Branch off a freshly-fetched origin/main (a stale local main can revert newer origin commits on squash). pr-auto-merge.yml arms auto-merge with AUTO_MERGE_PAT (a plain GITHUB_TOKEN merge won't chain the deploy event) on every non-draft PR except codie/*, kevin/*, and feature/*, which need manual review. Safety lives in CI, not in the branch name.

  • deploy.yml's concurrency: deploy-production with cancel-in-progress: false queues simultaneous deploys instead of racing on the git index lock.
  • Docs-only changes (**.md) are paths-ignore'd — they never restart prod; a mixed code+docs commit still deploys.
  • Done means green: confirm live /api/v1/version short-SHA equals origin/main (Rule A) — a commit on main is not deployed until the workflow is green and the VPS confirms it.

How the project stays un-confusing (operating principles)

Six tenets that keep a large, multi-process, multi-scheduler agent platform legible to a newcomer.

PrincipleWhy it exists
1. Code is the source of truthDocs describe what the code does now; if they disagree, the doc is wrong. Citations are file.py::symbol — never line numbers (they rot on every edit) — and CI (scripts/doc_audit.py) mechanically enforces it.
2. The Platform Status Registry is the single indexOne place answers "what do we have, and is it live (vs parked / paused / retired / scaffolding)?" Read it first, before deep-diving any subsystem.
3. Status is read from code, not prose"Default-flip blindness" — a flag flips and the surrounding prose silently goes stale — is the #1 doc-rot pattern. So every rottable status is sourced from a machine-authoritative function, cited inline.
4. LLM-native intelligenceCode collects and preserves evidence and gates execution (safety, budgets, dedup, irreversible actions); the LLM synthesizes meaning over a bounded context. Don't build Pythonic pseudo-reasoning where a strong LLM with the right context will do.
5. Runtime-vs-dev contractUA runs on the VPS (uaonvps, user ua, always-on); the desktop is only Kevin's interactive cockpit. Anything continuous or scheduled is a deployment/systemd/ unit shipped via the pipeline — never a desktop --user timer.
6. Two schedulers + the migration-artifact gotchaRecurring work runs on two independent schedulers: ~20 systemd timers (deploy-independent) and 4 live in-process crons. A migrated job shows enabled:false in cron_jobs.json — that is the correct migrated state (the timer is the sole firer), NOT "off". Disambiguate with systemd_migrated_jobs.py::is_migrated_to_systemd.

The throughline is one authoritative source per question, mechanically defended. Behavior is defined by code; the registry is the one status index; CI audits the docs against the code they claim (via each doc's code_paths frontmatter, reverse-looked-up at change time). When you change behavior, you update the doc that owns it in the same PR — never deferred to the nightly sweep, which is a backstop, not the mechanism. This is what lets a newcomer trust a single answer instead of reconciling three drifting ones.

  • When something looks "dead," check the canonical resolver before concluding — an enabled:false cron, a private :8092 worker, and a parked-but-flag-gated lane all look dead from the wrong vantage point.
  • To verify anything live, read from code/runtime: the proactive report for fleet state, systemctl list-timers for schedules, and /api/v1/version for the deployed SHA — not a dated doc.
