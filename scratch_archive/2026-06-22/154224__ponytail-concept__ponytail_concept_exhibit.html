Bloat vs a blind cut vs a smart cut

The same over-loaded coding request, asked three ways, run on a local model on the desktop GPU (qwen2.5-coder:7b) · 2026-06-22

The experiment

One deliberately over-stuffed request is sent three times — identical task, three different system prompts — and we measure what each produces. The task tempts the model to over-build: "a config loader with hot-reload, YAML/JSON/TOML/INI, env overrides, schema validation, and a remote etcd sync client." Most of that is speculative — a real service just needs to read config at startup.

1 · Over-bloat

BASELINE

"Build it all, professionally. Don't omit anything."
No pressure to restrain. The model dutifully builds every requested subsystem plus scaffolding it imagines it needs. This is the 20K-token over-build the whole pattern exists to prevent.

2 · Naive cut

CAVEMAN

"Be brief. Short. As few tokens as possible."
Brevity with no judgment. It slashes output hardest — but it's truncation, not design: it compresses and drops things blindly, with no reasoning about what's actually needed. Cheapest, but you can't trust what it cut.

3 · Smart cut

PONYTAIL LADDER

"For each feature: needed? does stdlib do it? what's the minimum?"
Forces reasoning before code. It "spends" tokens thinking out loud — so it's not the fewest tokens — but it justifies every cut and emits the tightest, most defensible code of all three.

Flip to the thing that matters — the size of the code you have to own and read at 3am — and PONYTAIL produces the least of all three, even though it used nearly 2× CAVEMAN's tokens. It reasoned its way to the minimum instead of truncating into it.

The point

Fewest tokens is not the goal — the right minimal code is.

Over-bloat (BASELINE) builds everything you named, needed or not. A naive cut (CAVEMAN) just makes the answer shorter — it's cheap but blind, and you can't trust what it quietly dropped. The ladder (PONYTAIL) pays a little extra to reason first — necessity, then "does the standard library already do this," then the minimum — and lands on the tightest, most defensible code.

That's the lazy-senior-engineer move: not "write less," but "decide what genuinely needs to exist, then write only that." The few hundred extra tokens of reasoning are the cheapest insurance you can buy against either over-building or silently under-building.

Numbers are from a live local-GPU run (qwen2.5-coder:7b on a GTX 1660 Ti) — deltas are vs BASELINE. The exact ratios shift by model and run; the shape — bloat > smart-cut > blind-cut on code size — is the durable lesson.
