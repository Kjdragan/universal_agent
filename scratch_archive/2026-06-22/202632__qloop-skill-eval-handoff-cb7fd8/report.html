3. QLoop APPLICABILITY ASSESSMENT (honest — do not overclaim)

What QLoop is: an evaluator-controlled refine loop for non-deterministic artifacts (reports, websites, agent workflows). A pure deterministic controller (qloop/controller/decision.py::decide) owns the stop/pivot decision via a 7-rule first-match ladder over (EvaluationResult, ControllerPolicy, history, iteration) → accept/refine/pivot/reset/human_review. The LLM judge only recommends; the controller decides (its recommended_action is consulted in EXACTLY ONE rule, #6 reset). Two packages, one-way dep (qloop Core never imports qloop_harness; test-enforced). Text judge = Z.ai/GLM raw SDK; vision judge = Gemini. run_critique() = single-shot scorer of an arbitrary artifact; run_build_loop() = full multi-iteration ladder.

Where QLoop does NOT fit: - Skill-triggering eval is near-binary detection, already solved by evaluate_triggering. QLoop adds a stop/pivot ladder over a yes/no signal. Fit: WEAK. - Boundary disqualifier: QLoop's text judge (providers/claude_sdk_zai.py::complete) is the raw AsyncAnthropic SDK (messages.create) despite "sdk" in its name — no claude_agent_sdk/claude -p/subprocess, so the judge cannot invoke skills. (Nuance: claude_sdk_zai.py::ClaudeSdkZaiProvider.runtime_env() does exist for Agent-SDK/subprocess builder calls, so QLoop's generation side is NOT skill-blind — the disqualification is judge-path-specific, not whole-system.) Either way, QLoop's judge can't host UA skill-triggering eval. - Skill-description tuning is a proven weak vein; QLoop would optimize a rubric proxy that doesn't predict the real (triggering) outcome. Don't build it.

Where QLoop SHINES (the real opportunity, and UA's genuine gap): - Subjective quality judging of reports / demos / websites. UA generates these and judges them once or not at all on quality. run_critique(artifact_type, text=|url=, critique_only=True, mock=...) is a drop-in programmatic scorer; the deterministic controller supplies the standing refine loop UA lacks. Fit: STRONG. - Visual quality. The Gemini vision judge grades rendered UI design (any URL or local PNG) — UA has NO visual grader; would catch silent render failures (the mermaid footgun). Fit: STRONG (standalone, needs only GEMINI_API_KEY). - ZAI provider boundary. QLoop's never-mutate-os.environ + never-cross-vendor discipline is cleaner than UA's for the raw-SDK case, but not interchangeable — UA must mutate env for the CLI/Agent-SDK path and Max OAuth, which QLoop has no path for. Fit: PARTIAL — borrow patterns, keep build_inference_env.

UA already wanted this (key discovery): urw/ (orchestrator.py::URWOrchestrator, evaluator.py::CompositeEvaluator) is a COMPLETE evaluator-controlled loop (binary+constraint+LLM+sub-agent judge, replan, human-review, confidence ladder) — but DORMANT, reachable only via the --urw CLI flag (main.py::_run_urw_from_cli), wired into NO heartbeat/cron/Task Hub (only main.py + agent_core.py reference it outside urw/). So the cheapest path may be grafting QLoop's missing pieces (pivot/reset split, calibration, discrimination-gate, vision judge) onto URW rather than greenfield. Caveat: URW's LLMJudgeEvaluator uses raw AsyncAnthropic with the same OAuth fragility we found in the bundled optimizer, and is NOT routed through batched_judge (no FUP/fail-closed protection).

6. GOTCHAS (will bite you)

  • QLoop default branch is master; .venv/site-packages and .claude/worktrees/ are STALE copies — canonical source is src/qloop + src/qloop_harness.
  • run_critique runs with history=[], iteration=1, so stagnation / twice-failed-gate / max-iterations rules CANNOT fire single-shot — only run_build_loop exercises the full ladder.
  • The evaluator's recommended_action is consulted in EXACTLY ONE rule (#6 reset); the controller can override it. "Evaluator decides, controller rubber-stamps" is the WRONG mental model.

  • Subjective-criterion pass/fail in the router is DERIVED/informational; the controller owns dimension floors. Acceptance flips False only on failed deterministic/interactive criteria.
  • Opus model default: QLoop's LIVE .env sets ANTHROPIC_DEFAULT_OPUS_MODEL=glm-5.2 — same as UA. Only the committed .env.example is stale at glm-5.1. (Do NOT expect a divergence.)
