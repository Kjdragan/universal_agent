Desktop GPU for local-Ollama demos

Why the Ponytail demo stalled, the real fix (it runs on the GPU now), the standard model + guards, and the email-approval gate. · 2026-06-22

1 What triggered this

Simone's rescue-evaluator did exactly its job: it independently reproduced a blocked demo and handed back a clean verdict. The ponytail-yagni-ladder-ollama demo was correctly built but could not be verified — its 3 live refactor passes run gemma3:12b at ~7 tok/s on CPU, so a full run takes 15–20 min and every timeout in the system (the 600s no-progress watchdog) killed it mid-pass.

The verdict assumed the only option was a faster/smaller model or buying GPU infra. But there was a wrong premise underneath it.

2 Root cause: the GPU and the demo live on different machines

The demo runs on the VPS, which has no GPU and never will. The 6GB GPU is on the desktop. They're decoupled — a desktop GPU can't accelerate VPS inference. So the honest fix isn't "buy GPU for the VPS"; it's run GPU-bound demos on the desktop, and make every demo small enough to fit a 6GB card.

3 Verified: the demo runs end-to-end on the GPU

Installed Ollama user-local on the desktop (no root), pulled the standard model, and ran the full 3-pass demo on the GTX 1660 Ti. All three passes completed — the run that stalled forever on the VPS finishes in ~4 minutes.

4 Standard model + the 6GB guards

Picked qwen2.5-coder:7b as the standard for local-GPU demos: best code quality that stays under 5GB. The new provision-local-gpu-ollama skill provisions the box and refuses (not warns) anything that breaks the limits you set.

5 The GPU-demo approval gate default OFF

Most demos still build on the VPS. A demo that needs a local GPU is detected, parked, and you get an email — nothing builds on the desktop unless you say so. Contract-safe: nothing runs autonomously on the desktop; it only acts when you drive it interactively.

1
Detect & park VPS
The demo sweep flags a GPU-bound demo, queues it so CODIE never auto-claims it, and stamps gpu_approval: pending.
2
Email you VPS
A specific explanation (what the demo is, why it needs a GPU, VPS-CPU vs desktop-GPU estimate, the model) + an HMAC-signed Approve link.
3
You click Approve YOU
The always-on VPS gateway verifies the signature, records approved, and shows the exact command to run.
4
Build on the desktop DESKTOP
You run /gpu-demo-build <task> interactively → it provisions the GPU model via the skill, builds, and marks the task complete.

Master switch UA_GPU_DEMO_DESKTOP_APPROVAL_ENABLED (off by default → prod unchanged until you flip it). No desktop daemon, poller, or timer anywhere in the flow.
