Does our loop system beat bare /loop?

A controlled A/B on glm-5.2 — bare self-judged loop vs our improved system (discipline + independent voted judge), graded by an objective arbiter (pytest / regex / a blind independent grader), never the loop's own judgment. 2026-06-22.

Headline: on every fair task, the improved system tied the bare loop — and the improved arm spent 4× the model calls to do it. The one apparent win was an information confound, not the independent judge. For glm-5.2 on clear tasks, the voted-judge machinery did not earn its cost.

The confound — and why it matters

First pass, N2 looked like a decisive win: bare 0.25 → improved 1.00. But the arms weren't fair: bare was only told "make it compelling," while the improved arm was handed the exact 4 grading criteria (no buzzwords / name a user / quantify / ≤60 words). So the gap was information, not mechanism — bare was graded on rules it never saw.

Fix: give both arms the same explicit criteria, so the only difference is self-judging vs independent voted judging. Result across 4 trials: bare 1.00 = improved 1.00. The gap vanished entirely. Bare glm-5.2, once told the rules, self-verifies them perfectly.

What this means

For glm-5.2 on clear, well-specified tasks, our improved loop system adds no measurable benefit over a bare loop with a clear prompt — and the independent voted judge costs ~4× the model calls for it. The only thing that ever helped was making the check explicit — which is the discipline's "state the CHECK" rule, i.e. good prompting, not the judge machinery.

Why the judge didn't help (and when it might)

  • The judge's whole rationale is catching a loop that self-declares done while actually short. glm-5.2 on these tasks wasn't self-deceived — it was accurate about its own completion.
  • A same-model judge shares the author's blind spots, so its upside is narrow: it mostly catches "looks-done vs meets-stated-criteria" — which is better solved by stating the criteria up front.
  • Where it could still pay off (untested here): genuinely ambiguous quality a model can't self-assess, a weaker model that rushes, or a different-model judge (cross-model, different blind spots).

Honest limits

One model (glm-5.2); small, well-specified tasks; 1 trial on the nulls (4 on the decisive N2). The nulls are clean perfect ties, but harder/ambiguous tasks or a cross-model judge could move the result. This says "no benefit here," not "no benefit ever."

Recommendation

Keep the discipline, demote the judge. "Restate as GOAL / CHECK / STOP — make the fixed check explicit" is validated and cheap (it's the part that mattered): keep it in the /loop hook + CLAUDE.md. The independent voted judge should be a reserved, opt-in tool for genuinely ambiguous or cross-model verification — not the default, because on clear tasks with a capable model it's pure overhead.
