Network Assessment

Kevin's Network: how the desktop, the VPS, and the tailnet fit together

Assessment across universal_agent, ebook_reader (Tome), SonosKD, ensemble (ex-tv_art_gallery) & home_tray — plus a live read-only probe of both machines. Basis for a proposed network-description-for-kevin skill. Read-only; nothing was changed.

Bottom line. You've converged on a genuinely good, consistent architecture — a clear three-tier placement model (VPS = always-on prod, desktop = home-LAN anchor + cockpit, tailnet = the wiring & the auth boundary) with disciplined secrets and a "tailnet-membership-is-the-auth" stance. The pattern is sound enough to write down as one skill. Two things need attention regardless of the skill: Docker is publishing several services straight to the public internet past your firewall (Ollama & Jaeger are answering strangers right now), and the tv_art_gallery→ensemble rename left two live services running off deleted files — one reboot from silent failure.

The mental model an agent needs (three tiers)

Always-on runtime

VPS · uaonvps

The production host. If it must survive the desktop sleeping, run on a schedule, or be reachable 24/7, it lives here.

  • Deployed apps: universal_agent, Tome
  • Public web: nginx → clearspringcg.com
  • Private artifacts: /scratch via tailscale serve
  • Scheduling: systemd timers (not cron/routines)
  • No GPU. No home-LAN access.
Cockpit + LAN anchor

Desktop · mint-desktop

Interactive dev, plus the only place that can touch the house or use a residential IP.

  • Interactive Claude Code (Max plan)
  • Home devices: Sonos, LG/Denon/AppleTV, Matter
  • SSDP/multicast bridges (fauxmo → Echo)
  • Residential-IP egress (yt-dlp / YT Music)
  • GPU work (GTX 1660 Ti, Ollama)
Wiring + auth boundary

Tailnet · taildcc090

Connects the two machines and is the access control. Membership = the auth.

  • MagicDNS: uaonvps…, mint-desktop…
  • tailscale serve → private HTTPS
  • tailscale funnel → the rare public case
  • SSHFS path bridge (same path both ways)
  • ACLs by device tag (in-repo policy in UA)

The two decisions that drive everything

1 · Where should this run?

Does it touch home devices, SSDP/multicast, or need a residential IP?
→ Desktop, mandatory. The VPS physically can't reach the LAN or broadcast. (Sonos, ensemble, fauxmo, yt-dlp.)
Does it need a GPU?
→ Desktop. VPS has none, ever.
Must it be always-on / scheduled / public?
→ VPS. Desktop sleeps; VPS doesn't. Schedule = systemd timer.
Everything else (a web app, an API, an agent)?
→ VPS by default; desktop only for interactive dev.

2 · How do I make it reachable?

Private (just you, your devices)? — the good pattern
Bind 127.0.0.1:PORT, expose with tailscale serve. Gives TLS + a clean MagicDNS URL, off every public NIC. (UA scratchpad, Tome do this.)
Public web (customers)?
nginx on the VPS's public IP, with each listen 443 pinned to that IP so it never collides with tailscale serve. Never hand-edit — use the install script.
Genuinely public one-off?
tailscale funnel. Real internet exposure — audit it; don't leave it pointing at dead ports.
Anti-pattern: bind 0.0.0.0:PORT + raw http://host:PORT
Works over the tailnet but ships no TLS and sits on every interface. (SonosKD/ensemble do this today.)

What you're doing right (keep these — they belong in the skill as rules)

  • Consistent secrets story. Every repo bootstraps from Infisical with 3 machine-id creds in a gitignored .env; values never printed; device creds verified untracked.
  • Tailnet-as-auth, on purpose. "Membership is the auth" is stated and applied uniformly, not stumbled into.
  • Loopback + tailscale serve as the private-exposure gold standard (UA, Tome) — TLS, MagicDNS, off the public NIC.
  • The "never host SonosKD on the VPS" rule — correct reasoning about LAN + residential IP, written down so it isn't relitigated.
  • nginx IP-pinned to avoid the 443/tailscale collision; installed by script, not hand-edited.
  • Sound SSH posture: key-only, no password auth, ufw default-deny; SSHFS path guarantee for seamless cross-machine files.

Where it drifts from best practice

Prioritised. The top two are worth acting on soon; the rest are the raw material for the skill's "gotchas" section.

HIGH Docker publishes ports to 0.0.0.0, bypassing ufw — services are on the public internet right now

ufw shows default-deny and correctly blocks 22/8001/8002. But Docker writes its own iptables rules ahead of ufw's INPUT chain, so any ports: ["11434:11434"]-style mapping binds the public NIC regardless. Probing the public IP 187.77.16.29 from off-box confirmed live:

PortServiceState from the internet
11434OllamaOPEN, no auth — returned {"version":"0.30.10"} to a stranger
16686Jaeger UIOPEN — HTTP 200 (trace data browsable)
9000agent dev-UI (uvicorn)OPEN — 307 → /dev-ui/
3030, 4318dashboard, OTLP collectorOPEN — 200 / responding
6379Redisexposed but authed — -NOAUTH (password set ✓)

Ollama on a GPU-less VPS is also just questionable — worth asking why it's there at all.

Fix: in each compose file bind these to loopback or the tailnet IP — 127.0.0.1:11434:11434 (or 100.106.113.93:…) — so they leave the public NIC. Or add explicit DOCKER-USER deny rules. Don't rely on ufw alone with Docker.

HIGH The tv_art_gallery→ensemble rename left two live services running off deleted files

ensemble-fauxmo.service and tvag-art-screensaver.service are running right now from /home/kjdragan/lrepos/tv_art_gallery/… — a directory that no longer exists (merged into ensemble). They survive only because Linux keeps a deleted binary's inode alive for a running process. Worse, the fix isn't just the installed units: ensemble/deployment/fauxmo/fauxmo_config.json and the repo's own copy of the unit still hardcode the dead path, so even a clean restart breaks.

Fix: repoint the two systemd units + fauxmo_config.json's plugin path to /home/kjdragan/lrepos/ensemble/…, then restart once to prove it. You're one reboot from a silently-dead Alexa bridge and screensaver.

MEDIUM The private-exposure pattern isn't standardized across the home repos

UA and Tome do it right (loopback + tailscale serve, TLS). SonosKD and ensemble bind 0.0.0.0:PORT and are reached as raw http://…:8557/:8558 — no TLS, on every interface. ensemble's own HANDOFF flags "Tailscale Serve HTTPS (PWA SW)" as a still-open TODO, because service workers need HTTPS. SonosKD already half-fixed this (a serve on :8443) but still also binds raw.

Fix: standardize home services on loopback + tailscale serve. One pattern, TLS everywhere, clean MagicDNS URLs, and nothing accidentally on a public NIC.

MEDIUM The SSHFS path bridge is unowned, undocumented OS substrate with three competing definitions

The bridge is live and working (verified BRIDGE-OK from the VPS). But the desktop side has three different mount mechanisms on disk — mnt-vps.service, mount-vps.service+/usr/local/bin/mount-vps-tailscale, and different target paths (/ vs /root) — none enabled, yet something mounts it at boot. UA's own networking doc already flags that no sshfs command lives in any repo; it's assumed substrate that can silently drift.

Fix: pick one mechanism, systemctl enable it, delete the other two, and check the chosen unit into a repo (or document it in the UA networking doc). Right now nobody owns the thing the "path guarantee" depends on.

MEDIUM Deploy discipline is bimodal — and an agent can't tell which repo is which

universal_agent has full CI/CD (PR→GHA→remote_deploy→health-gate→/api/v1/version check). Tome, SonosKD, ensemble are all manual git pull+restart with no CI gate, and a couple still carry raw scp steps (unifetch RUNBOOK Phase 0) or an unused gh-clone bootstrap (ensemble). For personal scale that's fine — but the inconsistency is the risk: a fresh agent doesn't know the rules per repo.

Fix: the skill should carry a one-line "how this repo deploys" per repo (below), so an agent never guesses. Optionally graduate Tome to the same CI pattern.

LOW Stale public Funnel on the desktop pointing at dead ports

The desktop has tailscale funnel ON exposing /→:8000, /support→:8002, /contracts→:8003 to the public internet — but only :8002 has anything listening; / and /contracts are dead. Funnel is real internet exposure (unlike serve), so it deserves a periodic audit.

Fix: drop the funnel paths you aren't using; keep only what genuinely needs to be public.

LOW Root-keyed SSHFS + root SSH still enabled; LAN IPs hardcoded as fallbacks

The VPS→desktop mount authenticates as root with allow_other; root SSH is key-only but still enabled now that ua has sudo. And 192.168.4.28/.103 are sprinkled as fallback defaults across ensemble's drivers and SonosKD's seed IP.

Fix: move the bridge to a non-root user; consider disabling root SSH; centralize the LAN IPs into one config value per repo. All minor.

The proposed skill: network-description-for-kevin

One skill that hands any agent — dropped into any of your repos — the mental model above, so it places work correctly and reaches it correctly without re-deriving your topology. It's a reference/description skill, not an action skill: it teaches, it doesn't run commands.
