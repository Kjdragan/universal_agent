Infisical → GitHub — Assessment & Implementation Plan

For the universal_agent CI/CD · 2026-06-23 · evidence-based on our actual GitHub secrets

Recommendation: Adopt — App Connection + Secret Sync

Let Infisical own the CI secrets it already owns at runtime.

We have 14 GitHub Actions secrets, and at least 5 of them are the same values we already store in Infisical, mirrored into GitHub by hand. That hand-mirroring is a silent drift surface: rotate the value in the vault and CI keeps the stale one until someone remembers to update GitHub. An Infisical GitHub App connection + Secret Sync makes the vault the single source of truth and pushes those values into GitHub automatically. Low effort, real payoff, matches exactly what you described.

1. Two opposite features (don't conflate them)

The Infisical docs you sent describe two things that point in opposite directions. Getting them straight is half the decision:

A · GitHub Actions integration (pull)

The Infisical/secrets-action runs inside a workflow and pulls secrets from Infisical at job runtime, exposing them as env vars. Auth is OIDC — GitHub mints a short-lived token, no Infisical creds stored in GitHub.

GH workflow→ OIDC →Infisical→ env vars →job steps

B · GitHub App Connection + Secret Sync (push)

Infisical connects to GitHub via a scoped GitHub App and pushes vault secrets into GitHub Actions secrets (repo / environment / org). One-way, Infisical → GitHub, kept in sync on change. This is the "inject variables into GitHub from Infisical" you described.

Infisical→ push →GitHub Actions secrets→workflows read ${{ secrets.X }}

They can even be alternatives: B pre-populates GitHub's secret store so workflows read ${{ secrets.X }} exactly as they do today; A skips GitHub's store and fetches live. For our deploy, B is the smaller change because our workflows already consume ${{ secrets.* }}.

The win in one line: at least 4–7 of these secrets are duplicated by hand between Infisical and GitHub. Today, rotating one in the vault leaves CI stale and silent. A Secret Sync makes that rotation propagate to CI automatically — that's the whole pitch, and it's real.
Bootstrap chicken-and-egg: INFISICAL_CLIENT_ID/SECRET are how the workflow would authenticate to Infisical in the first place, so they can't usefully be synced from Infisical to bootstrap themselves. Either leave those two in GitHub as-is (simplest), or remove them entirely via OIDC (phase 2, §5). Everything else is freely syncable.

3. What to adopt, and what to skip

  • ✔ Adopt — GitHub App Connection + Secret Sync (B), scoped to only the universal_agent repo, pushing the sync-eligible subset above into repo Actions secrets. This is the direct fix for the drift surface.
  • ◐ Optional, phase 2 — OIDC via secrets-action (A), purely to delete INFISICAL_CLIENT_ID/SECRET from GitHub (replace two long-lived stored creds with short-lived OIDC tokens). Security hardening, not functionality. Defer until B is proven.
  • ✘ Skip — using A to fetch all runtime secrets into the workflow. The workflow doesn't need them; the VPS fetches runtime secrets itself at service start. Pulling the full set into CI would just widen blast radius.

4. Implementation plan

Phase 0 · Confirm the sync set must
  • For each sync-eligible secret, confirm the value is identical in Infisical (production env) and GitHub. Resolve any drift now — the vault value becomes canonical.
  • Verify the Tailscale secrets exist in the vault; if not, add them (or leave them GitHub-native).
  • Decide naming: GitHub secret names must match what workflows reference (${{ secrets.AGENTMAIL_API_KEY }} etc.). The sync must preserve exact key names.
Phase 1 · Create the GitHub App Connection must

In Infisical: Project → Integrations → App Connections → GitHub. Choose the GitHub App method (recommended over OAuth/PAT — finely-grained, least-privilege, better audit trail).

Grant the minimum permissions the docs specify:

Repository permissions:
  Metadata     : Read-only
  Secrets      : Read and write
  Environments : Read and write
  Actions      : Read
Install on:    Only select repositories → universal_agent  (NOT "All repositories")

Authorize the install scoped to the single repo. This is the security-critical step: scope it to universal_agent only so Infisical can never touch unrelated repos.

Phase 2 · Configure the Secret Sync must
  • Create a GitHub Secret Sync: source = Infisical production env at the path holding these keys; destination = universal_agent repository Actions secrets (not org-level).
  • Scope the sync to the explicit sync-eligible key list — do not bulk-push the whole vault into GitHub. Push only what CI reads.
  • Enable sync-on-change so a vault rotation propagates to GitHub automatically.
  • Run the initial sync; confirm with gh secret list that values updated (timestamps move) and names are unchanged.
Phase 3 · Verify end-to-end must
  • Rotate one low-risk synced secret in Infisical (e.g. re-enter UA_OPERATOR_TELEGRAM_CHAT_ID to the same value) and confirm the GitHub secret's Updated timestamp advances.
  • Trigger a no-op workflow run (or wait for the next deploy) and confirm CI still authenticates and the deploy goes green — proves names/values survived the sync.
  • Document the connection + sync in project_docs/06_platform/01_secrets_and_infisical.md (which secrets are vault-managed-into-GitHub, which stay GitHub-native, and the "don't hand-edit synced GitHub secrets" rule).
Effort estimate: Phases 0–3 are a single short session of UI configuration in Infisical + one verification deploy. No UA code changes required — workflows keep reading ${{ secrets.* }} exactly as they do now. This is config, not engineering.

5. Optional phase 2 — OIDC, kill the stored creds

Once the sync is trusted, the remaining long-lived creds in GitHub are INFISICAL_CLIENT_ID/SECRET. The secrets-action with OIDC removes them:

permissions:
  id-token: write   # GitHub mints the OIDC token
  contents: read

steps:
  - uses: Infisical/secrets-action@v1.0.9
    with:
      method: "oidc"
      identity-id: "<machine-identity-id>"   # public, safe to commit
      project-slug: "<project-slug>"
      env-slug: "production"
  # subsequent steps read the fetched secrets as env vars
Caveat that limits the upside: our VPS still needs INFISICAL_CLIENT_ID/SECRET for its own startup bootstrap — it runs outside GitHub's OIDC context. So OIDC removes the creds from GitHub, but the workflow would then have to fetch them from Infisical (via OIDC) and hand them to the VPS over the existing stdin channel. Net: two fewer long-lived secrets in GitHub, slightly more workflow plumbing. A genuine but modest hardening — do it only after B is solid, and only if removing stored creds from GitHub is worth the added step.

6. Risks & guardrails

  • Over-broad GitHub App scope. The one mistake that matters. Install the App on selected repositories → universal_agent only, never "All repositories," never org-wide secrets unless a concrete need appears.
  • Don't sync the bootstrap creds into themselves. Exclude INFISICAL_CLIENT_ID/SECRET from the sync set (chicken-and-egg). Handle them via "leave as-is" or OIDC.
  • Hand-edits to synced secrets will be overwritten. Once a secret is sync-managed, GitHub is a read-replica — edit it in Infisical, not in GitHub. Document this so no one is surprised when a manual GitHub edit reverts on the next sync.
  • Name discipline. A sync that renames or namespaces keys would break ${{ secrets.AGENTMAIL_API_KEY }} references. Keep destination names identical to the vault keys.
  • Blast radius is bounded. This is one-way Infisical→GitHub for a small, explicit key set on a single repo. It does not touch the VPS runtime path, the loader, or deploy mechanics — so it's safely reversible (delete the sync + connection to revert to manual secrets).
