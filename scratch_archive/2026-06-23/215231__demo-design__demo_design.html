/demo — point it at anything, get a briefing + a working demo

Design synthesis from our grilling session · 2026-06-23 · all decisions below are resolved unless marked OPEN

What it is. An interactive desktop slash command. You point it at any source — a GitHub repo, an X/Threads post, a blog/news announcement, a YouTube video, a dropped-in image, or even a one-line topic — and it researches the thing until it understands it well enough to build with it, writes an operator briefing, then builds a real, runnable demo that exercises the capability end-to-end. The lasting payoff is a skill — a modular capability our agents can adopt.

It is mostly wiring existing parts together behind a generalized front door, plus one genuinely new piece: a per-source repo.

The flow

SEED
Point at a source
URL · repo · post · image · topic + optional one-line steer
→
RESEARCH
Investigate outward
The seed is a starting point. Find the blog/docs/repo/tutorial until the API surface is understood with current ids/signatures.
→
BRIEF
Briefing + plan
Publish to scratchpad & hand you the link immediately, then keep going. Interruptible, never blocks.
→
BUILD
Build the demo
Archetype-routed (below). Real endpoint, no mocks, small suite of 2–3 examples.
→
VERIFY
Run & iterate
Execute, evaluate, FEEDBACK.md, re-run. Several retries, no wall-clock timeout.
→
LAND
Repo + register + skill
/new-repo → GitHub + VPS, register on the demo dashboard, stage the skill as a promotion candidate.

Only hard-stops on a genuine blocker (no API access, source simply isn't buildable) — and even then it ships the briefing + an honest "couldn't verify because X" markup.

What the demo builds — two archetypes

Archetype 1 · agent capability

A dialogue-driven agent + a custom skill

New Claude Code feature, an MCP, an agent tool, a new API param — anything that makes an agent more capable.

  1. Proof script — make the capability actually work against the real thing (fastest path to "it runs").
  2. Extract a skill via skill-creator — refactor the proven functionality into an optimized, modular skill.
  3. Base agent + demonstration dialogue — scripted text sets a premise/task → triggers the skill → it runs its process (multi-turn if needed) → the feature is demonstrated.

The agent pursues an outcome by reaching for the skill — not running a hardcoded script. The base agent is one reusable scaffold (override allowed for unusual topologies).

Archetype 2 · general framework

A straight script that drives the framework

A new library / model / API — your "Google OCR parser" example. Build in that stack, first-class.

  1. Proof script — use the framework to accomplish one real task, end-to-end, against the real thing.
  2. Small suite — a couple of examples covering the feature's main uses.
  3. Graduate (optional) — if it's worth lifting into the agent toolbox, run the same skill-creator step and it becomes an agent capability.

Local-GPU / local-model demos run on the desktop Ollama (qwen2.5-coder:7b) — the only place they can run.

The synthesis: the skill is the universal durable artifact regardless of source type. The script→skill bridge means even a general-framework demo can graduate into a modular agent capability. Every /demo grows the toolbox.

Open — my recommendations, awaiting your sign-off

A · Input surface — what can you point it at?
Rec: any URL (repo / X·Threads / blog·news / YouTube / docs) + a local file/image you drop in + a short freeform topic ("the new Claude Code tag feature") + an optional one-line steer ("focus on the streaming part"). The seed can be thin; research fills the rest.
B · Briefing form
Rec: durable markdown in the repo (CONCEPT.md + README.md, reusing the teaching-doc pattern) and a rendered scratchpad page. Default to auto-rendered markdown; upgrade to a hand-authored visual-explainer page when diagrams genuinely help. Always light mode.
C · Research stopping rule
Rec: stop when you can write correct code against the current API (verified ids/signatures) or when marginal info drops off; soft cap of distinct sources to avoid runaway. If still not buildable, ship the briefing + a clear blocker note.
D · Retry budget
Rec: up to ~3–5 iterate cycles before deferring with an honest blocker report — mirrors the existing evaluator loop. No wall-clock timeout; liveness-based progress only.
E · Repo naming
Rec: snake_case derived from the feature (e.g. claude_code_tag_feature_demo), private, lands at ~/lrepos/<name> on desktop + VPS via vps_bootstrap.sh.
