/demo — point it at anything, get a briefing + a working demo

Design synthesis from our grilling session · 2026-06-23 · every decision below is resolved — ready to turn into an implementation plan

What it is. An interactive desktop slash command. You point it at any source — a GitHub repo, an X/Threads post, a blog/news announcement, a YouTube video, a dropped-in image, or even a one-line topic — and it researches the thing until it understands it well enough to build with it, writes an operator briefing, then builds a real, runnable demo that exercises the capability end-to-end. The lasting payoff is a skill — a modular capability our agents can adopt.

It is mostly wiring existing parts together behind a generalized front door, plus genuinely new pieces: a per-source repo, a self-improving demo_factory, and a recall vault.

What this is — and isn't. /demo is a portable Claude Code slash command, installable into ~/.claude/ and runnable in any session on any repo — not a feature inside Universal Agent. We author it from the UA session and lean on UA's code as proven patterns to replicate, never as a dependency to import.

Packaging split: dragan-plugins holds the portable surface — the /demo entrypoint (next to /new-repo) + graduated skills; the demo_factory repo holds the engine + data — base SDK template, growing local skill library, vault, lessons, regression suite, verifier/hook scripts. The command is a thin launcher that drives the factory. Promotion pipeline: a skill is born in a demo → accrues to the factory's local library → (proven + reused) → graduates into dragan-plugins, becoming a global capability.

Research helpers — required, routed by domain. The research phase delegates to specialists rather than relying on training-data recall (critical for features announced today). The claude-code-guide agent — doc-grounded against official Anthropic docs + web search — is the required helper for any Claude Code, Agent SDK, or Claude API source, i.e. the common archetype-1 case. Others by domain: gemini-api-dev / Context7 / deepwiki for third-party frameworks; the YouTube transcript skill when the best explainer is a video.

Plus (additive, not a replacement): for source-heavy topics, dump the gathered sources into a NotebookLM notebook and query it grounded — a richer aggregator layered on top of the routed-specialist research above, never instead of it.

What the demo builds — two archetypes

Archetype 1 · agent capability

A dialogue-driven agent + a custom skill

New Claude Code feature, an MCP, an agent tool, a new API param — anything that makes an agent more capable.

  1. Proof script — grounded by the claude-code-guide agent's doc research, make the capability actually work against the real thing (fastest path to "it runs").
  2. Extract a skill via skill-creator — refactor the proven functionality into an optimized, modular skill.
  3. Base agent + demonstration dialogue — scripted text sets a premise/task → triggers the skill → it runs its process (multi-turn if needed) → the feature is demonstrated.

The agent pursues an outcome by reaching for the skill — not running a hardcoded script. The base agent is one reusable scaffold (override allowed for unusual topologies).

Archetype 2 · general framework

A straight script that drives the framework

A new library / model / API — your "Google OCR parser" example. Build in that stack, first-class.

  1. Proof script — use the framework to accomplish one real task, end-to-end, against the real thing.
  2. Small suite — a couple of examples covering the feature's main uses.
  3. Graduate (optional) — if it's worth lifting into the agent toolbox, run the same skill-creator step and it becomes an agent capability.

Local-GPU / local-model demos run on the desktop Ollama (qwen2.5-coder:7b) — the only place they can run.

The synthesis: the skill is the universal durable artifact regardless of source type. The script→skill bridge means even a general-framework demo can graduate into a modular agent capability. Every /demo grows the toolbox.

Adopted from the fast convergence lane

/demo is the intensive marquee build; the existing autonomous tutorial_build lane is the quick, automatic one. Worth lifting from the quick lane into this one:

Mechanical completion checks (venv / uv.lock / pyproject present + README run section) → cheap objective gate before "done"
SOURCES/ grounding → research artifacts become cited reference docs the build + briefing draw from
Buildability judgment (repurposed) → becomes the un-demoable decision: graceful degradation, not a gate
Skipped: daily ceiling / approval gate — autonomous quota control, irrelevant to an interactive build

/goal as the build-loop driver — verified, with guardrails

Critical multi-lens analysis (code-grounded, ultracode). Verdict: adopt with hardening. The reframe: UA's "internal goal loop" IS native /goal — claude_cli_client.py::_run_goal_loop_mission literally shells claude -p "/goal <condition>". Using it in /demo is consolidation onto a production primitive, not a new mechanism (and when goal-eligible, UA already disables its own retry loop). The cody-work-evaluator pass/iterate/defer loop is a different, autonomous, cross-subprocess beast — not imported into /demo.

Decisive rule: /goal is the loop DRIVER (foreman), never the completion GATE (judge). Its evaluator is transcript-only (no tools, no file reads). The gate stays a deterministic verifier that actually executes the demo and prints one forge-proof line — DEMO_VERIFY: PASS demo_id=… ran=1 exit=0 endpoint=anthropic_native …; the /goal condition only greps for that token. On the desktop the evaluator is real Haiku (Anthropic Max), not the autonomous ZAI glm-4.5-air rubber-stamp — so it's a better fit here than autonomously.

Hooks are first-class here — not banned

The "vanilla settings" rule was always about not inheriting the production daemon's ZAI-routing env/hooks (so a demo proves a clean real-Anthropic endpoint) — never a ban on hooks as a capability. Confirmed in code: POLLUTION_INDICATORS strips the inherited hooks key, not the disableAllHooks toggle — hooks stay enabled (/goal itself IS a Stop hook). For /demo we keep endpoint integrity the right way — the Anthropic-Max launch profile + per-build endpoint_hit verification — not by stripping hooks.

So: the improving base SDK template owns a curated, version-controlled hook set; new hooks are added & tested in the template's own test environment (conflicts contained to one evolving source), recorded in a HOOKS.md ledger, with a template-CLAUDE.md guideline to minimize hooks unless they earn their place — record every one, never overuse, never prohibit. Hooks actively serve this design: the verifier-gate is a deterministic script Stop hook that prints DEMO_VERIFY; per-skill lesson capture (Loop 1) is a PostToolUse hook.

Recursive improvement — the compounding engine

The proven pattern is your own /new-repo lessons loop (blocker → root cause → fix → file, folded back, buffer reset). We apply it at three levels, all hosted in the dedicated demo_factory repo (base scaffold + shared skill library + meta-lessons + regression suite).

Loop 1 · per-skill

Every authored skill ships LESSONS.md — written at creation & at runtime, read at the start of every run, folded into the skill body on consolidation. "Next time it runs, it's already solved."

Loop 2 · capability accrual

Every demo skill auto-registers into the factory's shared skill library (local to the demo-builder, not broad UA). The builder gets monotonically more capable; repeat capabilities are reused, not rebuilt.

Loop 3 · process memory

The /demo skill + base scaffold accrue meta-lessons about building demos (provider key names, "X-type sources are briefing-only," model-currency notes), read at the start of every build.

Guard / signalWhy it's there
Reuse → promotion signalA skill reused across N demos auto-ranks higher as a broad-UA promotion candidate (decision #7). Reuse count is a better adoption signal than judging a skill in isolation.
Demos = regression suiteEvery demo is runnable + verified, so when a shared skill folds in a lesson, re-run the demos that use it. The safety rail that stops self-modifying skills from silently rotting.
Structured + boundedReuse the new-repo lesson schema (actionable checklist, not prose); dedupe → fold durable lessons into the body → reset the buffer so logs never bloat.

Skill librarian — keeps the library from sprawling. The vault pre-flight doesn't just answer have-it / don't: when it finds related skills, an LLM judgment picks reuse · expand an existing skill · or create-new-then-flag-for-consolidation. A periodic librarian pass then consolidates a crowded cluster into a master skill (HyperFrames video-ad-producer shape — one skill, internal modes) or a skill group (stitch shape — directory + README + sub-skills), and re-runs the regression suite. Lean: a skill-group template + LLM judgment, not deep infra. Raises the promotion bar (decision #7).

Knowledge vault — the recall layer

State of the existing vault: the claude-code-intelligence machinery exists but is empty & parked (no pages since Apr 28; population loop flag-off; NotebookLM output never ingested back). The one store that actually works is scratch_archive. So we build fresh-and-lean in demo_factory on that proven pattern rather than revive drifted machinery.

One entry per capability (not per demo run — re-visiting a capability extends its entry). Each unifies what /demo already emits: the concept brief, the visual briefing, the proven skill(s), the demo repo link, the lessons, and the failure notes. Index = the scratch_archive pattern (jsonl ledger + browsable HTML) + SQLite FTS so an agent can actually ask "have we covered X?" — stdlib, deterministic, no embeddings infra.

Failures are first-class. When a concept relates to a real skill but won't demo (needs hardware we lack, a paid tier, an unreleased API, or it's simply not demoable), the process writes "what it is + why it didn't demo + what it'd take" into CONCEPT.md and the visual briefing, and the vault marks that entry un-demoable — not dropped. The durable form of "best-effort, mark honestly."

Payoff · future demos

A pre-flight query — "do we already have a concept / skill / known-failure for this?" — kills re-research and surfaces a reusable skill. The recall half of the loops above; without it they accumulate but can't be retrieved.

Payoff · real project builds

A searchable library of "how to actually use capability X," each entry backed by a verified runnable demo + a liftable skill + known gotchas. A genuine engineering asset, not demo exhaust.

Consumption & connection layer — all core, these are what turn a pile of demos into a living asset:

"Where could we use this?" → each briefing ends with an LLM section mapping the capability to our existing projects/repos. The biggest connector from factory to real work.
NotebookLM audio overview per capability → listen to a new-capability briefing on the go (paper-to-podcast / gemini-tts-narrator).
Wiki, repurposed → NotebookLM generates a per-capability-cluster overview on demand into the vault entry, instead of the dead autonomous card→wiki loop.
Weekly "what's new in the toolbox" → LLM synthesis over new vault entries; emergent themes + newly-possible skill combinations.
Cross-capability composition demos → once the library has several skills, build demos that combine them; showcases emergent value + tests interop.

(NotebookLM-as-research-aggregator lives in the research layer above — additive to it, not here.)

Settled details (A–E — all approved)

A · Input surface — what can you point it at?
Approved ✓ any URL (repo / X·Threads / blog·news / YouTube / docs) + a local file/image you drop in + a short freeform topic ("the new Claude Code tag feature") + an optional one-line steer ("focus on the streaming part"). The seed can be thin; research fills the rest.
B · Briefing form
Approved ✓ durable markdown in the repo (CONCEPT.md + README.md, reusing the teaching-doc pattern) and a rendered scratchpad page. Default to auto-rendered markdown; upgrade to a hand-authored visual-explainer page when diagrams genuinely help. Always light mode.
C · Research stopping rule
Approved ✓ stop when you can write correct code against the current API (verified ids/signatures) or when marginal info drops off; soft cap of distinct sources to avoid runaway. If still not buildable, ship the briefing + a clear blocker note.
D · Retry budget
Approved ✓ up to ~3–5 iterate cycles before deferring with an honest blocker report — mirrors the existing evaluator loop. No wall-clock timeout; liveness-based progress only.
