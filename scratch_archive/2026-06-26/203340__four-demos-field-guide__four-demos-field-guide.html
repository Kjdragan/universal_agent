Four Demos: Convergence & Orchestration

A field guide to telling autoresearch-loop, self-correcting-loop, convergence-pipeline, and subagent-orchestration apart.

Reconstructed from each repo's README.md + SKILL.md · 2026-06-26

The thing that makes these easy to confuse: three are the same shape — iterate one artifact against one judge until it converges — and they differ only in what the judge is. The fourth is unrelated: it splits one task across many parallel workers and merges them.

The 3 + 1 structure

autoresearch-loopjudge = a number → hill-climb to "best"
self-correcting-loopjudge = pass/fail → repair to "green"
convergence-pipelinechains the two: optimize, then verify

Convergence family #1#2#3

One artifact, one evaluator, iterate to convergence. Temporal: improve over rounds. #3 is literally #1 followed by #2.

Orthogonal #4

subagent-orchestration — not a loop. Spatial: split one task across N parallel subagents in their own context windows, then a synthesizer merges. Fan-out → fan-in.

1 · autoresearch-loop the optimizer

autoresearch-engine /dragan:autoresearch-loop

Kjdragan/demo-autoresearch-engine · a framework-agnostic port of karpathy/autoresearch

A self-driving hill-climb. Give it one mutable artifact, a read-only evaluator that returns one comparable scalar, and a proposer. It loops: propose → apply → score → keep-if-better / revert-if-worse → log to results.tsv → repeat until the score plateaus. The artifact is left holding the best variant; the ledger is the full audit trail. Use it when "better vs worse" is a number and there's no single right answer, just more optimal.

Judge
scalar number (min or max)
The move
keep-better / revert-worse (coordinate descent, one knob/step)
Stops when
score plateaus (patience)

2 · self-correcting-loop the verifier / repairer

self-correcting-loop /dragan:self-correcting-loop

Kjdragan/demo-self-correcting-loop · "loops, not prompts"

Generate a candidate → run a real, executable rubric (pytest, jsonschema, a compiler, a linter) → if it fails, feed the actual failure output back and revise → repeat until it passes or a round cap. The rubric is ground truth and is hidden from the generator, so the model satisfies the check, not the test text. Use it when correctness is pass/fail and you can run the check.

Judge
executable rubric (pass/fail)
The move
revise, conditioned on the real failure
Stops when
the rubric passes (or round cap)

3 · convergence-pipeline the composition

optimize-then-verify /dragan:convergence-pipeline

Kjdragan/demo-optimize-then-verify · chains #1 and #2 (vendored verbatim — genuine reuse)

The other two, chained — not reimplemented. Stage 1 runs autoresearch-loop to hill-climb a draft toward a scalar score; Stage 2 seeds self-correcting-loop with Stage 1's winner and repairs it against an executable rubric until it passes. Order matters: get something robust first, then make it exactly correct. Use it only when a task has both a soft metric to maximize and a hard check it must ultimately pass.

Judge
scalar → then pass/fail
The move
optimize, then repair the winner
Stops when
the verifier passes

4 · subagent-orchestration the odd one out

subagent-orchestration /dragan:subagent-orchestration

Kjdragan/demo-subagent-orchestration · Claude Agent SDK fan-out → fan-in

Nothing to do with convergence. A Claude Agent SDK orchestrator declares named subagents in code (ClaudeAgentOptions.agents=), fans one bounded task out to several of them in parallel (multiple Agent tool-use blocks in one turn, each its own clean context window), then a synthesizer merges the results. The skill carries four hard-won traps: recursion fork-bomb isolation, decoupling the file-write from the model, Anthropic-vs-ZAI auth routing, and proving the fan-out actually happened (≥2 distinct subagents) rather than trusting exit-0. Use it when one task has independent facets to handle concurrently and then combine.

Judge
a synthesizer (the merge agent)
The move
dispatch N workers at once → merge
Stops when
all workers return + synthesis written

The fastest decision rule

Is your judge a number or a pass/fail check?

→ Number → autoresearch-loop.   → Pass/fail → self-correcting-loop.   → Both → convergence-pipeline.

Not iterating at all, but splitting work across parallel workers? → subagent-orchestration.
