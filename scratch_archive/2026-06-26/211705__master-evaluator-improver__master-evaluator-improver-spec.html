Master Evaluator / Improver — Design Brief

A single skill that judges, gates, and improves the interim steps of a long-running process — auto-selecting a deterministic or non-deterministic path. Built by composing four existing loop skills.

Self-contained handoff for an AI coder · grounded in four shipped repos · 2026-06-26

Why we're building this — the purpose

One master skill for "generate → evaluate → gate → improve"

We want a single master evaluator/improver skill that, given any step in a longer flow, figures out how that step should be judged and drives it to a passing state. Rather than a human hand-picking which loop to use, the skill itself decides the path — deterministic (a runnable pass/fail check) or non-deterministic (a subjective judgment) — and runs the right kind of evaluation/improvement.

Its core job is to facilitate long-running code or processes that move through interim, gated checkpoints. Each interim step must pass its evaluation — deterministic or not — before the flow proceeds; or, when we simply want a step better before moving on, it improves it in place. It's the judging/gating layer inside a generator → evaluator → improver loop.

The ordering is the point. Try to pass the gate as cheaply as possible first; only escalate into optimization when the gate fails — then retry the gate. Improvement is the expensive fallback, never the default. That efficiency ordering is a first-class part of the design.

1The building blocks — four existing loop skills

The master skill does not reinvent anything; it routes among and composes four loops that already exist as standalone skills/repos. Three are the same shape (iterate one artifact against one judge until it converges) and differ only in the judge; the fourth is a parallel primitive.

Optimizer

autoresearch-loop

/dragan:autoresearch-loop · demo-autoresearch-engine

Self-driving hill-climb (port of karpathy/autoresearch). One artifact, one comparable scalar evaluator, a proposer. Loop: propose → apply → score → keep-if-better / revert-if-worse → log to results.tsv → repeat until the score plateaus. Use when "better vs worse" is a number.

Verifier

self-correcting-loop

/dragan:self-correcting-loop · demo-self-correcting-loop

"Loops, not prompts." Generate → run a real, executable rubric (pytest, schema, compiler, linter) → on failure feed the actual error back and revise → repeat until it passes or a round cap. The rubric is ground truth and hidden from the generator. Use when correctness is pass/fail.

Composition

convergence-pipeline

/dragan:convergence-pipeline · demo-optimize-then-verify

Chains the two (genuine reuse, not reimplemented). Stage 1 hill-climbs a draft toward a scalar; Stage 2 seeds the repair loop with that winner and fixes it against an executable rubric until it passes. Use when a task has both a soft metric and a hard check.

Parallelism

subagent-orchestration

/dragan:subagent-orchestration · demo-subagent-orchestration

Claude Agent SDK fan-out → fan-in. Declares named subagents in code, dispatches N in parallel (each its own context window), then a synthesizer merges. Carries four traps (recursion isolation, decoupled write, auth routing, proof-of-fan-out). The parallel primitive — for breadth and, crucially, as a judge panel (§4).

2The methodology — gate-pass first, optimize-on-fail

The efficiency ordering (the core rule)

Don't optimize what already passes. Run the gate first. If it passes, proceed. Only when it fails do you spend tokens on improvement — and you escalate in cost-ordered tiers, retrying the gate after each. Breadth (parallel fan-out) is the last resort, not the first.

One fact that shapes the design: what the optimizer does with "many facets"

The autoresearch engine optimizes one artifact against one scalar, walking the knobs by coordinate descent (one knob/step, keep-if-the-global-score-improved). A multi-knob config is already a many-facet search, and the single global score sees the interactions between knobs — coupled facets are handled correctly, just sequentially. So you only reach for parallel fan-out when facets are genuinely separable (own artifact + own score) or to run parallel restarts of the same loop to escape local optima — not as a generic "make it more optimized" knob.

The escalation ladder

Note that convergence-pipeline already is self-correcting-loop with an optimize stage in front — so escalating from #2 to #3 isn't a second repair pass; it's giving the repairer a stronger starting draft.

Gate first · free-ish

Run the gate

Generate the step, run its evaluation. Pass → proceed immediately. No optimization spent.

Tier 0 · cheapest repair

self-correcting-loop alone

Gate failed: repair against the (deterministic or judge) gate for K rounds, conditioned on the real failure.

Tier 1 · draft too weak

convergence-pipeline

Repair stalled: optimize a better draft toward the soft metric first, then re-verify the hard gate.

Tier 2 · stuck or big/separable

subagent-orchestration wraps Tier 1, N-wide

Separable → one pipeline per facet. Coupled → parallel restarts. Merge, then re-verify the merged result, and keep the branch that converges.

The merge is the dangerous step. When you fan optimization across facets and recombine, a slightly-coupled merge can score worse than any branch or fail a gate none failed alone — a naive fan-in manufactures regressions. That's why the executable verify runs last, on the merged artifact, and a failed verify loops back into another fan-out with the real failure fed in. This is what makes "keep working until the gate passes" real.

3Path routing — deterministic vs non-deterministic

(It can detect this from the criterion itself — is a runnable command supplied? is there a numeric target? is the goal prose-quality?)

4The non-deterministic path — turning a noisy judge into a trustworthy gate

The loops assume a deterministic, comparable signal. A subjective goal ("more persuasive", "higher answer quality", "better design") has neither a runnable check nor a clean scalar. The good news: only the evaluator changes — the loop machinery (keep/revert, ledger, round cap, proposer) is already fine, and the LLM proposer is supposed to be non-deterministic.

The core problem

Swap in an LLM-as-judge and you inherit three failure modes: it scores the same artifact differently across runs (variance); the proposer learns to flatter the judge (reward hacking); and there's no absolute ground truth. Keep/revert needs a trustworthy comparison — so every fix below makes the judge stable enough to gate on.

AdaptationWhat it doesWhy it works
1 · Pairwise, not absoluteAsk "is B better than current-best A?" not "score B 0–10".LLM judges are far more reliable head-to-head. Maps directly onto autoresearch keep/revert.
2 · Judge panel = use #4 as the evaluatorFan the evaluation out to N independent judges (different lenses/rubrics), aggregate by vote/median.The elegant combination: subagent-orchestration becomes the variance-reduction layer. An ensemble cuts judge noise.
3 · Variance-aware keep/revertKeep only if the panel prefers B by more than the judge's measured noise floor (re-judge the same pair to estimate it).Stops the loop "accepting" changes inside the judge's own jitter — the #1 way a subjective hill-climb drifts forever.
4 · Rubric decompositionSplit "quality" into sub-criteria (clarity, accuracy, completeness, tone); score each; aggregate.Several narrow judgments are more stable and more auditable than one holistic number.

Guards against hill-climbing the judge's biases

  • Hold-out / rotating judges. Optimize against one judge framing; accept with a different one. A train/val split for subjectivity.
  • Anchor on whatever is deterministic. Even subjective artifacts usually have hard constraints (length, required facts, valid format, banned claims). Gate those deterministically (the hybrid path); judge only the soft remainder.
  • Human-in-the-loop as periodic ground truth. Without an objective signal you only ever reach "judged to pass," never "proven." An occasional human spot-check on the ledger recalibrates the judge and catches drift cheaply — a natural fit for the interim-gate checkpoints.

5The master loop — facilitating a long-running, interim-gated flow

This is how the skill plugs into a generator/evaluator/improver process. For each interim step of the larger flow, it routes, gates cheaply, escalates only on failure, records the decision, and proceeds. It supports both modes you described: gate-to-proceed (the step must pass) and polish-before-proceed (improve a step we simply want better).

for each interim step→generate candidate→route the evaluator (§3)→run the gate
pass →record · proceed to next step
fail →Tier 0 repair→Tier 1 optimize+verify→Tier 2 fan-out+merge+verify→ re-gate
# Master Evaluator/Improver — pseudocode (composes the four loop skills)
def run_step(step, mode="gate"):              # mode: "gate" (must pass) | "polish" (improve then confirm)
    candidate = generate(step)
    path, gate, scorer = route(step.criterion)   # §3: deterministic | scalar | subjective | hybrid

    # --- efficiency ordering: try to pass before improving ---
    if gate(candidate).passed and mode == "gate":
        return record(step, candidate, "passed-first-try")

    # --- escalate in cost-ordered tiers, re-gating after each ---
    for tier in [
        lambda c: self_correcting_loop(c, gate),                     # Tier 0  (#2)
        lambda c: convergence_pipeline(c, scorer, gate),             # Tier 1  (#1→#2 via #3)
        lambda c: fanout_then_merge_then_verify(c, scorer, gate),    # Tier 2  (#4 wraps #3)
    ]:
        candidate = tier(candidate)
        if gate(candidate).passed:
            return record(step, candidate, "converged")

    return record(step, candidate, "unconverged", escalate_to_human=True)

# subjective gate() is a judge PANEL (§4): pairwise vs best, N judges, keep only past the noise floor.
# hybrid: scorer optimizes the soft objective; gate() is always the deterministic hard check.
# every step appends to a ledger — the audit trail of interim gate decisions for the whole run.

#1 the loop + ledger   #2 the deterministic gate/repair   #3 the optimize-then-verify frame   #4 breadth and the judge panel. The master skill is the router + tier-escalator + ledger around them.

6Honest limits

  • Cost. A judge panel multiplies eval cost by panel size every iteration; Tier 2 multiplies by N. The gate-first ordering exists precisely to avoid paying this when a step already passes. Pairwise (vs current-best only) keeps judging O(1) per step.
  • Fuzzy convergence on the subjective path. "Stops improving" becomes "the panel stops preferring new candidates" — needs a noise-tolerant plateau and patience, or it chases jitter.
  • Preference, not proof. The subjective ceiling is "the panel and the rubric agree it's better." Where being wrong is expensive, keep a deterministic verify as the final word and a human on the recalibration loop.
  • Routing can misclassify. If the router picks "subjective" when a cheap deterministic check existed, it burns tokens; prefer the most deterministic path the criterion supports, and let the criterion declare its check when it has one.
