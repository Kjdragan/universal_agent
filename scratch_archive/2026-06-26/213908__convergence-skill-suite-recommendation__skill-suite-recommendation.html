Convergence Skill Suite — Portfolio Recommendation

How to name, structure, and trigger a suite of generate→evaluate→gate→improve skills so the right one is selected automatically — individual primitive or full chained master.

Validated against official Claude Code docs (via claude-code-guide) + skill-creator trigger discipline · 2026-06-26

What you asked for

A portfolio, not a monolith: keep the individual loops usable on their own (most work only needs one), but also have a master that chains them — and let Claude select the right altitude per situation, whether that's a single primitive or an entire gated flow.

You flagged a need for two master modes, and I agree: a single-artifact "dev tool" that drives one process to optimal/passing, and a multi-step orchestrator that gates the interim steps and final outcome of a long-running flow (e.g. inside /dragan:demo). Conclusion up front: yes, build both.

One naming caveat from the docs. Claude picks skills purely from their descriptions, and the docs warn that closely-related descriptions collide. A shared token like "loop" across several names worsens that — so I dropped "loop" everywhere except autoresearch-loop (which you wanted to keep). If you prefer your pass-fail-correcting-loop, that's fine, but lean on disjoint verbs in its description to compensate. Names are cosmetic; descriptions do the real triggering work (§3).

The selection rule (this is what makes the suite work)

You know which loop you want → call the primitive (or optimize-then-verify if you know it's soft+hard). Claude auto-selects these from their narrow descriptions.

You don't know the eval path, it's multi-objective and needs deciding, or it's a gated multi-step flow → call a master, which routes. Gate-pass first; optimize only on failure; re-gate.

3Trigger discipline — the draft descriptions

This is where the docs say to spend the effort: with seven semantically-overlapping skills, the description field is the entire basis for selection, and overlap is the documented failure mode. Below are draft descriptions engineered for disjoint trigger terms and explicit downward deferral. (Final pass: run skill-creator's run_loop.py trigger-eval optimizer — 20 should/shouldn't-trigger queries, 3× each — to tune them empirically before shipping.)

autoresearch-loop

Tune ONE artifact toward a single numeric score until it plateaus — hill-climb / coordinate-search a config, prompt, or function against a measurable metric (loss, accuracy, latency, coverage, robustness %). Triggers: "optimize overnight", "auto-tune", "hill-climb", "make this measurably better" when better-vs-worse is a NUMBER. Not for pass/fail checks → pass-fail-repair; not for subjective quality → judge-panel.

pass-fail-repair

Drive ONE artifact to pass a runnable check — generate, run the real check (pytest, schema validator, compiler, linter), feed the actual failure back, revise, repeat until it passes or a round cap. Triggers: "loop until the tests pass", "fix until it compiles/validates", "repair against a real check". Not for a numeric score → autoresearch-loop; not for subjective quality → judge-panel.

subagent-fanout

Run ONE bounded task across several named subagents in parallel (separate context windows), then merge with a synthesizer — fan-out → fan-in. Triggers: multi-lens review, multi-angle research synthesis, parallel candidate generation, "several specialists then a merge". Also the substrate judge-panel is built on. Not for sequential single-artifact iteration.

judge-panel

Rate or compare an artifact on subjective quality with an LLM judge panel — N independent judges, PAIRWISE preference, majority/median vote, noise-floor margin for stability. Triggers: persuasiveness, helpfulness, design/prose quality — judgment with no runnable check and no clean number. The non-deterministic evaluator. Not for runnable checks → pass-fail-repair; not for numeric metrics → autoresearch-loop.

optimize-then-verify

Produce ONE artifact that is both robust AND correct by chaining: hill-climb a draft toward a numeric score, THEN repair the winner against a runnable check until it passes. Triggers: "make it good, then make it correct", "tune for a metric but it still has to pass the tests". Use when you ALREADY KNOW it's soft+hard. If the eval path is unknown/needs deciding → converge; single objective → the matching primitive.

converge  (master · single artifact)

Drive ONE artifact to a finished, passing state when the right evaluation path isn't obvious — auto-routes between numeric-optimize / pass-fail-repair / subjective-judge / hybrid, tries to PASS THE GATE cheaply first, escalates into optimization only on failure, then re-gates. A development optimizer: "get this to done / drive this process to pass / decide how to judge this". If you already know the exact loop, call that primitive instead.

converge-flow  (master · multi-step)

Orchestrate a long-running, multi-step process with interim gated checkpoints — each step must pass its evaluation (deterministic or subjective) before the flow proceeds, plus a final-outcome gate; or polish a step in place before moving on. Routes each gate via converge, keeps a ledger of interim decisions. Triggers: gated build pipelines, staged code generation, demo builds, "evaluate whether each interim step and the end outcome were achieved". The heaviest skill — usually invoked explicitly.

4Construction rules — verified against the official docs

From the claude-code-guide pass over code.claude.com/docs/en/skills + plugins/sub-agents pages. These shape the build:

✗ There is no "skill calls skill" API. Composition happens through Claude's reasoning or a skill body that names the next skill in prose — OR at the code level (an engine importing another engine, as optimize-then-verify already does). The masters are playbooks + a Python engine, not programmatic skill-callers.
✗ "Fires only on the compound case and defers" is NOT enforceable — no precedence engine. You get deferral via (a) sharply disjoint descriptions, (b) explicit "use X instead when…" text, and optionally (c) disable-model-invocation: true to make a master manual-only (/dragan:converge-flow). Recommended: primitives auto-trigger; converge-flow (the most over-fire-prone) leans manual or tightly trigger-tuned.
✓ Directory name = the command. A dir pass-fail-repair/ in the dragan plugin becomes /dragan:pass-fail-repair automatically — never hand-prefix dragan:. Frontmatter name is display-only (except a plugin-root SKILL.md). Use kebab-case.
✓ Parallel fan-out is a subagent capability, not skill execution. subagent-fanout / judge-panel must run real subagents (the SDK fan-out engine you already built), documented/triggered by the skill — the SKILL.md runtime doesn't parallelize by itself.
✓ Don't duplicate the shared loop engine. Hoist the common Python engine to the plugin root and reference it from each skill via ${CLAUDE_PLUGIN_ROOT} (confirm the exact var in plugins-reference). Keep each SKILL.md < ~500 lines; push detail into references/, executables into scripts/.

5The two master modes — and the /demo fit

You were right that both are needed; they're different scopes, and the multi-step one calls the single-artifact one per step.

Yes — converge-flow fits /dragan:demo directly. The demo factory already runs the build under a native goal loop with a deterministic verifier gate and a dead-air watchdog. converge-flow generalizes exactly that: each interim build stage (research → brief → scaffold → build) becomes a gated step; "demo runs & DEMO_VERIFY: PASS" is the final-outcome gate; and a judge-panel gate can add the non-deterministic check the factory currently lacks — "is the demo actually good?", not just "does it exit 0?". That makes the demo factory the ideal first real multi-step gated flow to harden converge-flow against, rather than building it in the abstract.

6Recommendation & sequencing

Build the suite as a layered portfolio, masters as thin orchestrators — and build both master modes, but in sequence, hardened on real work rather than speculatively.
  1. Rename the three existing skills + add judge-panel. Cheap, immediate clarity. judge-panel is the one genuinely missing primitive and is reusable on its own (any "evaluate this subjectively" need) — build it first regardless of the masters.
  2. Build converge (single-artifact dev tool). Mostly code: a router (det / scalar / subjective / hybrid) + the gate-first tier escalation, composing the existing engines. Testable in isolation; high standalone value as a dev optimizer.
  3. Build converge-flow against /dragan:demo. Let the demo factory's real gated stages shape the router, the ledger format, and the interim/final gate semantics. The masterpiece earns its complexity by surviving a real flow.
  4. Tune every description with the trigger-eval loop. 20 should/shouldn't-trigger queries per skill, run_loop.py, pick by held-out score. This is the step that prevents the seven-skill collision the docs warn about.
  5. Operate on-the-fly in the interim — for any chained case before the masters exist, follow the design brief as the written playbook so the gate-first discipline is consistent even un-codified.

Open risks to watch

  • Trigger collision / over-orchestration — seven overlapping skills; a master firing where a primitive would do. Mitigations: disjoint descriptions, deferral text, disable-model-invocation on converge-flow, and the trigger-eval tuning in step 4.
  • converge vs optimize-then-verify overlap — both touch single-artifact soft+hard. Keep the line crisp: optimize-then-verify = "I KNOW the path"; converge = "DECIDE the path / drive to done".
  • Shared-engine wiring — confirm ${CLAUDE_PLUGIN_ROOT} in plugins-reference before hoisting; don't copy the engine into each skill dir.
  • Premature abstraction — don't build converge-flow until the /demo (or another real gated flow) is the thing pulling it into existence.
