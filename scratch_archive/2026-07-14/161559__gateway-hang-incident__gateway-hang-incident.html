Production gateway event-loop hang

2026-07-14 · surfaced by a "YouTube Ingest Failed" email · recovered · durable fix landing

TL;DR. The "[ERROR] YouTube Ingest Failed" email was a symptom, not the disease, and had nothing to do with the YouTube OAuth re-auth (that succeeded). The real problem: the universal-agent-gateway process was hung — alive and listening, so systemctl reported "active", but its single asyncio event loop was blocked and it was serving nothing. The ingest POST to the local gateway just timed out at the transport layer, which the client logged as ingest_request_error. Restarting the gateway restored service instantly; a code fix prevents recurrence.

How it surfaced vs. what it actually was

The ingest client (hooks_service.py::_call_local_youtube_ingest_worker) POSTs to http://127.0.0.1:8002/api/v1/youtube/ingest. All 10 attempts returned ingest_request_error with an empty detail and http_status: 0 — that error is raised only in the except branch when the HTTP request throws a transport exception (no response at all). It is categorically not an auth failure (that path is ingest_http_error + a 401) and not a missing-transcript (that returns 2xx with ok:false). So the client couldn't reach a working server — pointing at the gateway, not YouTube and not the OAuth token.

Root cause. The py-spy stack of the event loop (MainThread):

list_personal_queue (task_hub.py — SELECT * FROM task_hub_items ...) dashboard_human_actions_highlight (gateway_server.py — async def endpoint) run_endpoint_function (fastapi/routing.py) ... → uvicorn event loop

The dashboard-widget endpoint dashboard_human_actions_highlight is declared async def, but its entire body runs synchronous, blocking SQLite work directly on the event loop: it takes the process-global _activity_store_lock (a threading.Lock) and runs list_personal_queue() against the shared ~1.1 GB activity_state.db. That table is only 2,310 rows, so the query is not slow by volume — it was blocked on a database lock. The lock is held cross-process by universal-agent-autonomous-runtime (Simone loops + the guarded VACUUM/wal_checkpoint(TRUNCATE) maintenance, which rewrite the 1.1 GB file under an exclusive lock). While that lock is held, every dashboard poll of this endpoint blocks the loop for the full 15s busy_timeout, then the next queued poll repeats — so the loop is frozen back-to-back for the whole maintenance window and the accept backlog grows until the gateway is wedged.

Why one endpoint can freeze everything

A FastAPI async def handler runs on the event loop. A synchronous blocking call inside it (a locked SQLite read) blocks the entire loop — no other request is served, no new connection is accepted, health checks time out. The codebase already learned this and moved other big-activity_state.db endpoints off the loop (see the "blocking the FastAPI" notes in gateway_server.py); dashboard_human_actions_highlight was a straggler that still ran on-loop — and, being a polled dashboard widget, it hit the locked window reliably.

Timeline & actions

  1. Detect. Ingest error email traced to a transport failure against the local gateway, not the OAuth token.
  2. Confirm hang. Accept backlog 318 & growing, CPU 0.7%, /api/v1/version timing out — event loop blocked.
  3. Capture forensics. py-spy dump of the blocked loop saved on the VPS (the exact stack above) before touching anything.
  4. Recover. sudo systemctl restart universal-agent-gateway → active, /api/v1/version = 200 in 2s, accept backlog 318 → 0. Service restored.
  5. Root-cause. Read the deployed code path + DB state (WAL mode, 15s busy_timeout, 2,310-row table, 1.1 GB file, cross-process VACUUM writer). Confirmed on-loop blocking read under the global lock.
  6. Durable fix. Convert the offending endpoint(s) from async def → plain def so Starlette runs them in its threadpool — a DB stall then degrades one request, not the whole loop. (PR landing; sibling-endpoint audit in progress.)

The fix. async def dashboard_human_actions_highlight → def dashboard_human_actions_highlight. The body has no await, so this is behavior-identical; the only change is that its blocking SQLite work now runs in Starlette's threadpool instead of on the event loop. The threading.Lock remains correct across threadpool workers. An audit is finding any sibling endpoints with the same on-loop pattern to fix in the same PR.
