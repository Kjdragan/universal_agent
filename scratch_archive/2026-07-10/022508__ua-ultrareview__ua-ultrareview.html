Unauthenticated destructive CSI purge — internet-reachable

securityLIVE
gateway_server.py:16424 — POST /api/v1/dashboard/csi/purge

The endpoint takes no Request and makes no auth call; it wipes CSI digests, activity-event notifications, the csi_specialist_loops table, and CSI task_hub_items. Sibling destructive routes (dashboard_discord_delete_*) all call _require_ops_auth; this one doesn't, and the only auth middleware matches /api/v1/ops/* only.

Proven live: GET /api/v1/dashboard/csi/digests on the running gateway returned HTTP 200 with 457 KB of data and no token, while the auth-gated control GET /api/v1/sessions returned 401. The destructive POST shares that same (absent) auth. Reachable publicly via nginx (app.clearspringcg.com) → Next.js → :8002, and with CORS allow_origins=["*"] + allow_credentials=True it is a cross-origin drive-by from any browser that can route to the host.

Fix: gate on _require_ops_auth(request) like the Discord siblings, or add a middleware covering /api/v1/dashboard/* and /api/v1/csi/*. Also tighten CORS off wildcard-with-credentials.

Idempotency ledger reads the wrong columns → duplicate external side effects

correctnessFable-confirmed
durable/ledger.py:253 — prepare_tool_call, UNIQUE-constraint race branch

On the concurrent-duplicate race the branch is meant to dedupe, it fetches the existing row with SELECT * (positional) and builds the receipt from the wrong indices: status=existing[14] is actually idempotency_key, idempotency_key=existing[13] is actually normalized_args_hash, and response_ref is hardcoded None — discarding the real cached response at index 18 (schema verified in migrations.py).

securityFable-confirmed

File-descriptor leak: journal handle never closed

reliability
agent_core.py:1990 — UniversalAgent.run_query

journal_file = open(journal_path, "a") is opened at the start of every turn, but the finally closes only run_log_file — there is no journal_file.close() anywhere in the file. The same UniversalAgent instance is reused across turns in a session.

Failure: on a long-lived process (gateway, Telegram bot) this leaks one fd per turn; eventually the process hits its fd ulimit and every subsequent file/socket open fails with "Too many open files" — a whole-process outage.

Fix: close journal_file in the same finally, or use a with/contextlib.ExitStack.

Blocking 180 s git-fetch outlives the 120 s mission lease → duplicate execution

correctness
vp/worker_loop.py:1536 — _provision_workspace

subprocess.run(["git","fetch",…], timeout=180) runs synchronously on the VP worker's single event loop. The lease is renewed once before _provision_workspace, but the recurring renewal task is created only after it returns; the lease TTL default is 120 s (vp_lease_ttl_seconds).

Failure: a slow-but-successful fetch (120–180 s) expires the lease with no renewal in flight; claim_next_vp_mission reclaims expired leases (status='running' AND claim_expires_at < now), so a worker restart (Restart=always) or a second instance duplicate-executes the coder mission — two worktrees, potentially two PRs.

Fix: offload the fetch with asyncio.to_thread, and/or start the renewal task before provisioning. Ensure fetch timeout < lease TTL.

Non-atomic Task-Hub claim / queue-rebuild / finalize

correctnesslatent
task_hub.py:2512 (claim) · :1894 (rebuild) · :4266 (finalize)

The runtime connection is autocommit (isolation_level=None) with no BEGIN IMMEDIATE and no UNIQUE(task_id) on assignments. Claim is check-then-act (status read, then UPDATE … WHERE task_id=? with no status guard); rebuild does an unconditional DELETE FROM task_hub_dispatch_queue then per-row autocommit inserts; finalize pops active_assignment_id without checking it owns the current in-progress state.

Failure (latent): two concurrent claimers double-dispatch a task; a concurrent rebuild wipes another's just-inserted rows and starves it; a late finalize clobbers a newer assignment's metadata. Currently masked by single-process, serialized, no-await dispatch — but fragile: one added await between check and update, or a second claiming process, reintroduces duplicate workers.

Fix: make the claim atomic (UPDATE … WHERE task_id=? AND status IN(...) and check rowcount, or BEGIN IMMEDIATE); add UNIQUE(task_id) for active assignments; scope rebuild/finalize to the owning id.

Action-link tokens never expire and are replayable

securitymedium
services/cron_artifact_notifier.py:1051 — sign_ack_token / sign_gpu_demo_token

Tokens are an HMAC over a static id/action string with no timestamp or nonce, so they never expire — yet the gpu_demo approve endpoint shows "Link expired or invalid" and overwrites decision state with no already-decided guard.

Failure: a leaked or forwarded approve/ack/promote email link stays valid forever and can be replayed to flip an approve↔reject decision the operator already changed. The codebase's own proactive_health_notifier.sign_finding_ack_token embeds and checks an expiry — that pattern just wasn't applied here.

Fix: embed an exp claim + verify it; mark tokens single-use (consume like the Link card-token flow).

perf / correctness
services/agentmail_service.py:4313 (time.sleep ≤3 s) · :2490 (subprocess.run gws, timeout=15 s)

Both run synchronously inside async inbound-email paths on the gateway's single event loop: a sqlite lock-retry sleeps up to ~3 s, and calendar scheduling shells out to gws for up to 15 s.

Non-constant-time secret comparison (CWE-208)

securitymedium
gateway_server.py:14503 (+session/ws/youtube auth) · hooks_service.py:4365

Ops-token issuance, the session API/WS auth, and the default webhook-token strategy all compare secrets with plain ==/!=. The constant-time hmac.compare_digest is used elsewhere in the very same modules (auth/ops_auth.py, the composio_hmac branch).
