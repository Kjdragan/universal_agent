Matt Pocock’s Skills v1.1 — Field Guide

A software-development lifecycle as agent skills: grill → spec → tickets → implement → review, plus the new Wayfinder for plans too big for one session. Installed and live-tested on this machine on 2026-07-11.

Source video: “New Skills! v1.1” (Matt Pocock) Repo: mattpocock/skills Release: v1.1.0 · 2026-07-08 Status: installed globally, stale names pruned, 4 live tests run

SetupInstall, update, prune

One installer command pulls the whole set. The CLI (skills on npm, by Vercel — the engine behind skills.sh) drops each skill into ~/.agents/skills or your agent's skills dir and wires it into Claude Code as a slash command. Skills load at session start — restart a session after installing. Note: the CLI phones home anonymous install telemetry by default (DISABLE_TELEMETRY=1 to opt out).

# install / re-sync everything (also how you pick up renames)
npx skills add mattpocock/skills -g -y -s '*' -a claude-code

# update previously-installed skills in place
npx skills update -g

# v1.1 renamed skills — the installer does NOT migrate old names; prune them yourself
npx skills remove to-prd to-issues decision-mapping -g -y
Rename trap. v1.1 renamed to-prd→to-spec, merged to-plan+to-issues→to-tickets, and renamed decision-mapping→wayfinder. The installer adds the new names but leaves the old ones behind — both answer until you prune. On this machine I removed to-prd, to-issues, decision-mapping, plus two older orphans (diagnose→now diagnosing-bugs, and zoom-out, deleted upstream).
Once per repo: run /setup-matt-pocock-skills in each repo before using the flow. It writes docs/agents/issue-tracker.md — which tracker to use (GitHub / GitLab / local markdown), the triage label vocabulary, and the “Wayfinding operations” section that code-review, to-tickets, triage and wayfinder all read.

The big pictureThe main flow

This is the answer to “what's the flow?” — the complete lifecycle v1.1 closes. Plan mode is replaced by an interview: the agent grills you, the answers become a spec, the spec becomes tickets, and each ticket is one fresh-context implementation session that reviews and commits itself.

ReferenceThe eleven skills that matter

user-invoked means you type it; model-invoked means the agent may reach for it on its own. HITL needs you in the loop; AFK runs alone.

/grilling model hitl

The shared interview primitive. Relentless questions about your plan until shared understanding.

  • One question at a time — “asking multiple questions at once is bewildering.”
  • Facts vs decisions — facts get looked up in the codebase; decisions are always put to you.
  • Confirmation gate — won't enact the plan until you confirm shared understanding.

/grill-me · /grill-with-docs user hitl

The two front doors to grilling. /grill-me is the bare interview; /grill-with-docs also captures ADRs and a domain glossary via /domain-modeling as you answer.

  • Replaces plan mode as the way work starts.
  • Both signpost up to /wayfinder when the effort won't fit one session.

/to-spec user

Turns the current conversation into a spec (née PRD) and publishes it to the repo's tracker. No interview — pure synthesis of what was already discussed.

  • Proposes the test seams (fewest, highest possible) and confirms them with you.
  • Template: problem → solution → extensive user stories → implementation + testing decisions → out of scope.
  • No file paths or code in the spec — they go stale. Exception: decision-rich prototype snippets.

/to-tickets user

Breaks a spec into tracer-bullet tickets — narrow vertical slices through every layer, each demoable, each sized for one fresh context window, each declaring what blocks it.

  • Quizzes you on granularity and blocking edges before publishing.
  • Wide refactors escape vertical slicing: expand–contract, migrating call sites in batches.
  • Work the frontier (unblocked tickets) one at a time with /implement.

/implement user

Deliberately tiny (15 lines): implement the spec/ticket, use /tdd at pre-agreed seams, typecheck often, full suite once at the end, then /code-review, then commit.

  • Exists so every ticket session starts the same way.

/tdd model

Now reference material, not a step list — hand it to an AFK agent and it just works. Red before green, one slice at a time.

  • Seams: tests live only at pre-agreed public boundaries.
  • Anti-patterns named: implementation-coupled, tautological, horizontal slicing.
  • Refactoring is no longer in the loop — it moved to code review.

/code-review model

Reviews the diff since a fixed point on two axes in parallel sub-agents: Standards (repo's documented rules + the Fowler smell baseline) and Spec (does the code do what was asked?). Never merges the two reports — one axis must not mask the other.

  • Fails fast on a bad ref or empty diff before spawning agents.
  • Repo standards override the baseline; smells are judgement calls, never hard violations.

/wayfinder user hitl

For a loose idea “too big for one session and wrapped in fog.” Charts the route as a shared map on the issue tracker: one wayfinder:map issue whose child tickets are the open decisions.

  • Plan, don't do — the map produces decisions, not deliverables.
  • Ticket types: research (AFK) · prototype (HITL) · grilling (HITL) · task.
  • Fog of war: only ticket what you can phrase sharply; the rest waits in “Not yet specified.”
  • One ticket per session; claim by assignment; blocking edges render the frontier in the tracker UI.

/research model afk

Spins up a background agent to investigate a question against primary sources — official docs, source code, specs — and leave one cited Markdown file where the repo keeps notes. You keep working while it reads.

/prototype model hitl

Throwaway code that answers a design question. Two branches: logic (tiny interactive terminal app to push a state model through hard cases) or UI (several radically different variations on one route).

  • One command to run, no persistence, no polish, state always surfaced.
  • When done: fold the decision into real code, park the prototype on a throwaway branch.

/setup-matt-pocock-skills user hitl

Run once per repo. Configures the three things every flow skill reads: which issue tracker (GitHub / GitLab / local markdown under .scratch/), the five triage labels, and where domain docs (CONTEXT.md, ADRs) live.

Also in the box assorted

The full set is ~39 skills: triage (issue state machine), qa (conversational bug filing), diagnosing-bugs, domain-modeling, codebase-design, handoff, writing-great-skills, teach, plus misc/personal ones (git guardrails, obsidian-vault, pre-commit setup…).

ReferenceThe Fowler smell baseline (code-review's cheat sheet)

Baked into every review whether or not the repo documents standards. Named after Refactoring ch.3 precisely because the terms are deep in every model's priors — naming the smell is enough to make the agent hunt it. Each is a judgement call, and a documented repo standard always overrides.

SmellFix
Mysterious NameRename; if no honest name comes, the design's murky
Duplicated CodeExtract the shared shape, call it from both
Feature EnvyMove the method onto the data it envies
Data ClumpsBundle the travelling fields into one type
Primitive ObsessionGive the domain concept its own small type
Repeated SwitchesPolymorphism, or one shared map
SmellFix
Shotgun SurgeryGather what changes together into one module
Divergent ChangeSplit so each module changes for one reason
Speculative GeneralityDelete it; inline until a real need shows
Message ChainsHide the walk behind one method
Middle ManCut it, call the real target direct
Refused BequestDrop the inheritance, use composition

ProofLive tests on this machine

I built a throwaway repo (orderly, a toy order-processing library) with a local-markdown tracker, deliberately planted defects, then ran the skills headlessly over GLM (the flat-rate backend, keeping Max usage untouched). Everything below is real output.

Test 1 — /code-review: does the Fowler baseline actually catch things?

I committed a discount-codes feature branch with seven deliberately planted defects — smells, standards violations, a missing spec requirement, and out-of-scope code — then ran:

9/9 planted defects caught, zero false negatives — plus two legitimate bonus findings I hadn't planted (Divergent Change: address formatters living in the discounts module; a duplicated reduce that should call the existing orderTotalCents). It also correctly passed the one requirement that was genuinely implemented (one-code-per-order) — no false positives. The closing line: “The two axes converge on proc2: it's the defect on both.”

Test 2 — /to-spec → /to-tickets: idea to dependency-ordered tickets

Scenario: a gift-cards feature whose decisions were “already discussed” (supplied as context). Two runs:

claude -p "/to-spec GIFT CARDS. Context from our discussion (decisions already made): ..."
claude -p "/to-tickets .scratch/gift-cards/spec.md — I pre-approve the breakdown ..."

What came out, all to the repo's local-markdown tracker:

  • A full template spec — problem → solution → 20 user stories (shopper, recipient, operator, developer perspectives) → implementation + testing decisions → out of scope. It obeyed the “no file paths, they go stale” rule (“a new gift cards module”), kept the agreed single test seam, and honestly recorded what it left open for the implementer (code-generation strategy, denomination set).
  • Three tracer-bullet tickets under .scratch/gift-cards/issues/ — issuance → redeem/balance → checkout integration — each a complete, demoable vertical slice with checkbox acceptance criteria and explicit Blocked by edges. Frontier: ticket 01.
  • A genuine catch: ticket 02 noticed the generated spec contradicted itself (zero-balance redemption: “applied is zero” vs. “clear error”) and wrote the reconciliation into the ticket rather than letting two agents implement it two ways.
Lesson learned the honest way. My first attempt mentioned “run /to-spec” mid-prompt instead of starting with it — and the agent correctly reported the command didn't exist, then improvised. That's because to-spec, to-tickets, wayfinder, implement, and grill-me are all user-invoked only (disable-model-invocation: true): the model can't see or trigger them — only your literally-typed slash command does. Type them; don't ask for them.

Test 3 — /wayfinder: chart a map for a foggy, oversized idea

Loose idea handed in: “Move orderly off in-memory/JSON data onto a real database. Maybe SQLite, maybe Postgres, unclear. Also unclear how discount codes migrate, and whether we need a migration tool for ~50k existing orders.” Since a headless run can't interview anyone, the grilling answers were supplied as a transcript (done: a locked decision set for /to-spec; constraints: solo dev, zero new services).

claude -p "/wayfinder — chart a map for this loose idea: ..."

It produced a textbook map at .scratch/db-migration/MAP.md and eight typed tickets with blocking edges — then stopped, exactly as the skill demands (“charting the map is one session's work”):

The subtler behaviors all landed too: six fog items parked in Not yet specified, each noting which ticket has to resolve before it can sharpen (cutover strategy, schema refinement, migration-tool design as distinct from its ticketed need…); implementation, gift cards and HA-Postgres explicitly ruled Out of scope with reasons; the repo's own coding standards pulled into the map's Notes as binding constraints on every schema decision; and the SQLite lean flagged as “a prior to be confirmed by 02/03, not a foregone conclusion.”

Test 4 — /research: background-agent reading legwork

Invoked the research skill in a live session with a question about the installer itself. It spun up a background agent (I kept working while it read), which returned a cited markdown file. Best finding — a correction to an assumption baked into my own prompt:

The `skills` CLI invoked by `npx skills add mattpocock/skills` is NOT Matt Pocock's tool.
It is the npm package `skills`, published by VERCEL LABS (repo vercel-labs/skills,
maintainers rauchg + quuu) — the engine behind skills.sh. Matt's repo is a skill
COLLECTION the CLI consumes: it shallow-clones the GitHub repo, discovers SKILL.md
files, and installs them (canonical store ~/.agents/skills + per-agent symlinks).

Telemetry: ON by default — GET to add-skill.vercel.sh/t on install/update/remove.
Disable: DISABLE_TELEMETRY=1 or DO_NOT_TRACK=1. Auto-disabled in CI.

Every claim in the full file carries its primary source (npm view output, the CLI's own dist/cli.mjs, README, skills.sh) — exactly the skill's contract.

House notesOn this machine specifically

  • Where things live. Canonical store is ~/.agents/skills with per-agent links; the v1.1 additions were installed as copies into ~/.claude/skills, which both Claude accounts share (~/.claude-acct2/skills is a symlink to it). Local house skills (crosstalk, sendit, the dragan plugin set) were untouched.
  • Pruned on 2026-07-11: to-prd, to-issues, decision-mapping, diagnose, zoom-out — all renamed or deleted upstream; verified against the repo's current tree before removal.
  • New sessions only. Skills are discovered at session start — restart Claude Code to see /wayfinder, /to-spec, /to-tickets, /code-review, /implement.
  • Per-repo setup required for the flow skills. code-review, to-spec, to-tickets, triage and wayfinder all read docs/agents/issue-tracker.md. Run /setup-matt-pocock-skills once in each repo you'll use them in (the test repo above shows what it produces).
  • Overlap warning. The house skill set already has neighbours for some of these (grilling vs the house grilling — now the same skill, updated; tdd likewise; research vs instant-topic-research-for-kevin; code-review vs the built-in /code-review command). When both could answer, name the one you mean.
