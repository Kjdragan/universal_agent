## Context

I'm continuing to build out the Discord integration for my Universal Agent (UA) system.

**Repository:** https://github.com/Kjdragan/universal_agent

**What's already done (Phase 1):**
The Discord Intelligence Daemon is built and deployed to production VPS. It:
- Runs as a systemd service (`ua-discord-intelligence.service`) using discord.py-self with my user token
- Passively monitors 912 channels across 28 Discord servers (45 total, 17 skipped)
- Layer 1: Stores all messages in SQLite with full metadata

- Layer 3: Scheduled batch LLM triage via ZAI (cheap model) producing structured insights
- Integrates with Task Hub (auto-creates missions), Simone (AgentMail alerts), and briefing pipeline (latest_briefing.md)
- All secrets loaded from Infisical (`DISCORD_USER_TOKEN`)
- Code lives in `discord_intelligence/` in the repo root

## Task 3: Phase 2 — Command & Control Server

**Read:** `discord_intelligence/HANDOFF_03_Discord_Command_Control.md`

Transform kdragan's Discord server into a structured operational interface for the UA system. This uses the BOT token (UA Disc Agent) since it needs to send messages, respond to slash commands, and post embeds.

### 3A: Create Channel Structure

Create these channels in kdragan's Discord server (the bot has Manage Channels permission):

```
📋 OPERATIONS
├── #simone-chat          — Direct conversation with Simone
├── #mission-status       — Auto-updated mission progress
├── #alerts               — Heartbeat issues, system health, urgent signals
└── #task-queue           — Current Task Hub state

🔬 INTELLIGENCE
├── #research-feed        — Layer 3 insights from monitored channels
├── #announcements-feed   — Layer 2 signals: releases, announcements
├── #event-calendar       — Upcoming events from monitored servers
└── #knowledge-updates    — New entries pushed to LLM Wiki

📦 ARTIFACTS
├── #briefings            — Daily/weekly intelligence briefings
├── #reports              — Generated research reports
└── #code-artifacts       — Code generated by agents

⚙️ SYSTEM
├── #bot-logs             — Technical operation logs
└── #config               — Configuration commands
```

### 3B: Implement Slash Commands

Register these as Discord application commands on kdragan's server (guild commands for instant availability):

**Operations:**
- `/status` — UA system overview (running missions, heartbeat health, queue depth)
- `/briefing [now|morning|weekly]` — Trigger or retrieve briefing
- `/task add <description> [priority]` — Create Task Hub item
- `/task list [status] [limit]` — Show task queue
- `/mission status [id]` — Active mission details
- `/mission list` — List all missions with status

**Intelligence:**
- `/research <topic>` — Commission ATLAS research mission
- `/discord search <query> [server] [channel]` — Search ingested Discord messages
- `/discord signals [type] [since]` — Show recent detected signals
- `/discord insights [limit]` — Show top unread insights
- `/wiki query <question>` — Query the LLM Wiki
- `/wiki add <title> <content>` — Add wiki entry

**System:**
- `/monitor add <channel_id> <tier>` — Add channel to monitoring
- `/monitor remove <channel_id>` — Remove channel
- `/monitor list` — Show monitored channels
- `/config triage-frequency <hours>` — Adjust Layer 3 frequency

### 3C: Wire Intelligence Feeds

Connect the daemon's output to the CC server channels:
- Layer 2 signals → auto-post to `#announcements-feed` as rich embeds
- Layer 3 insights → auto-post to `#research-feed` with relevance scores
- Scheduled events → auto-post to `#event-calendar`
- High-severity signals → also post to `#alerts`
- Daily briefing → post to `#briefings`

### 3D: Simone Chat Integration

Messages from the owner in `#simone-chat` get routed to Simone's processing pipeline. Simone's responses come back as embeds in the same channel. This creates a real-time alternative to email/AgentMail.

**How to identify the owner:** Use the owner's Discord user ID (351727866549108737, visible in the inventory data). Only route messages from this user ID.

### 3E: Thread-Based Mission Tracking

When a mission is created (by command or by intelligence trigger), auto-create a thread in `#mission-status`. Post updates to the thread as the mission progresses.

### Architecture Note:

Since the daemon uses the user token and the CC bot uses the bot token, they may need to be separate clients. Investigate whether discord.py-self can run both simultaneously, or if a second process is needed. If separate, create `cc_bot.py` alongside `daemon.py` with its own systemd service.

## Task 4: Phase 4 — Event Intelligence Pipeline (Investigation + MVP)

**Read:** `discord_intelligence/PRD_Discord_Event_Pipeline.md`

This is the most ambitious feature. Start with investigation, then build the MVP.

### 4A: Technical Investigation

Before building, investigate and document answers to:
1. **Can discord.py-self receive/record audio from stage channels?** What libraries are needed (ffmpeg, opus)? What are the actual capabilities?
2. **Discord TOS implications for recording** — What's the practical risk for recording public stage events?
3. **Google Calendar API** — Does our UA currently have Google Calendar integration? What's needed to add it? Can we use the Google Calendar MCP that's available in Claude?
4. **How do Discord Scheduled Events work in practice?** Do all servers use them? Or do some just post event announcements in text channels?

Document findings in a new file: `discord_intelligence/EVENT_PIPELINE_INVESTIGATION.md`

### 4B: Build the MVP (Text-Based Events Only)

Based on the investigation, build the MVP that handles text-based events (no audio recording yet):

1. **Event Discovery:**
   - The daemon already captures `on_scheduled_event_create` and `on_scheduled_event_update` — verify this works
   - Add text-based event detection in Layer 2 (messages mentioning events with time/date indicators)
   - Store discovered events in the `scheduled_events` table

2. **Event Notification:**
   - Post discovered events to `#event-calendar` in the CC server with rich embeds
   - Include: server name, event title, description, time (converted to CST/CDT — owner is in Houston), type
   - Add reaction-based owner input: ✅ (attend), 🎙️ (record), 📋 (summarize after), ❌ (not interested)

3. **Post-Event Text Digestion:**
   - After an event ends, gather all messages from the event's time window in the relevant channel
   - Run LLM digest: extract key takeaways, notable insights, action items, tools mentioned
   - Generate a briefing artifact (markdown)
   - Push to LLM Wiki as a knowledge base entry
   - Include in next morning briefing

4. **Calendar Integration (if feasible):**
   - Auto-create tentative Google Calendar entries for events above a relevance threshold
   - Owner can accept/decline from their calendar app

### 4C: Audio Recording (Stretch Goal)

If the investigation in 4A shows it's feasible:
- Bot joins stage/voice channel when event starts
- Records audio to file on VPS
- After event: transcribe with Whisper (UA has audio-to-text skill)
- Run same LLM digest pipeline as text events
- Store transcript and audio file, link in briefing
