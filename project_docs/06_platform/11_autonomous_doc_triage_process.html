Autonomous Doc-Drift Triage & Fix

nightly GLM detection → Opus triage → docs-only auto-merge · companion to ADR 10

The nightly sweep already finds documentation drift — but the issues it files are a write-only dead-letter that nothing reads. This is the process that closes the loop: a real-Opus agent that investigates each new drift issue, fixes what's genuinely wrong, and merges it behind a mechanical docs-only firewall. Option (a) — a desktop interactive Monitor — is live now.

02 · Three delivery options

Same triage brain and safety rails; only the trigger and the inference substrate change. (a) is chosen first; (b)/(c) are recorded migration targets.

OptionInferenceAlways-on?CostBest for
(a) chosen Desktop interactive loop Opus · Max (interactive — not the 6/15 metered credit) No — desktop-bound Max quota only Prototype / on-demand
(b) VPS API-key agent Opus · ANTHROPIC_API_KEY Yes Bounded API $ (~pennies–$/day) Production
(c) ZAI-detect + API-key-fix GLM detect / Opus fix Yes Lowest API $ Production, cost-min
Why not just use Max headless on a server? As of 2026-06-15 headless claude -p / Agent-SDK usage on a subscription draws a capped monthly "Agent SDK credit," then halts — which is why this repo already flipped Cody's default to ZAI on 2026-06-07. For always-on automation, an API key (Commercial Terms) is the policy-clean substrate; the Max subscription stays for interactive work — which is exactly what (a) uses.
