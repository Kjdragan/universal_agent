Canonical process · plat-doc-triage-automation · companion to ADR 11

The Autonomous Doc-Drift Dark Factory

nightly GLM detection → Opus triage on Max → docs-only auto-merge · proven live 2026-06-10

A nightly sweep finds documentation drift and files a GitHub issue. A Monitor wakes an Opus session running on the Max subscription; it triages each finding against real code, fixes what's genuinely wrong, stamps the doc, opens a PR, and auto-merges it behind a mechanical docs-only firewall — no human in the terminal. This page is the canonical map of that loop, the safety rails, the one structural limit we hit, and the live runs that proved it.

01 · End-to-end pipeline

The Monitor is just the doorbell — it wakes the loop on a new issue. Everything after runs in-session on pure Max OAuth (Opus), with no further polling needed.

03 · Triage classification — Opus vs the GLM that files the issues

The value over the GLM sweep is discipline: verify against real code, reject false positives, refuse to guess, and never re-fix a doc that's already correct.

VerdictMeansAction
REAL_DOCDoc genuinely disagrees with codeFix on a docfix/* branch + stamp last_verified
CODE_COMMENTDoc is correct; a stale .py comment/docstring contradicts the codeEscalate to backlog #906 — never touch .py from this loop
FALSE_POSITIVEInference artifact / truncated excerpt / doc already accurateNo change; note in the issue
UNVERIFIABLECited symbol not found in codeDefer — do not guess a fix
OPERATIONALA still-true env/infra fact not reflected in codeLeave — do not auto-edit
04 · The one structural limit (ADR §9) — and the hybrid path out

The first dry run surfaced this before we made the loop always-on.

A docs-only loop cannot converge on drift rooted in a stale code comment. PR #900 fixed 05_environments.md to say _HARDCODED_FALLBACK_MODE = "zai" (correct). The next sweep (#901) re-flagged it anyway — because cody_mode.py's own docstring still lists step-6 as "anthropic". The code contradicts itself, and the GLM blames the now-correct doc. The loop correctly refuses to thrash the doc and can't touch the .py file (the tripwire forbids it), so the finding recurs until the code comment is fixed.

Mitigations shipped: (1) last_verified stamping moves fixed docs out of the rotation; (2) code-comment findings escalate to backlog #906 instead of churning. Future hybrid (Option 3): when the backlog crosses ~10 items, add a comment/docstring-only-diff mechanical guard and a batched, human-gated Opus code PR — so inert comment fixes don't fire a prod deploy each, and the safe docs-only lane keeps running untouched.

Same triage brain + safety rails; only the trigger and the inference substrate change.

Operator decision (2026-06-10): the desktop is declared always-on — option (a) is the production posture, keeping the loop on credit-free Opus-on-Max interactive inference. Options (b)/(c) are contingency fallbacks only (ADR §7). The remaining gap is session lifetime: after a reboot or session exit, re-arm the issue watcher (ADR §10).
