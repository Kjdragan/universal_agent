# E2BIG Kernel Limits and Prompt Architecture

**Date**: 2026-03-22
**Domain**: Operations, System Architecture, Code Execution

This document establishes the architectural boundary conditions for building dynamic system prompts when executing agents locally via subprocesses, and details the failure modes that occur when those boundaries are breached.

## The Threat: Subprocess Prompt Bloat vs Linux Constraints

When the Universal Agent spawns a worker session (like the `claude` CLI via `subprocess.Popen`), the system prompt is passed as a command-line argument (`--system-prompt="{payload}"`). 

If this prompt grows too large, the execution engine will crash instantly with:
`[Errno 7] Argument list too long (E2BIG)`

### Why E2BIG Happens
There are two distinct limits enforced by the Linux kernel during `execve()`:

1. **`ARG_MAX`**: The total combined size of all arguments (`argv`) and environment variables (`envp`).
   - *Default*: Usually derived from `RLIMIT_STACK / 4`, which defaults to ~2MB.
   - *Our Fix*: We raised this via a systemd override (`LimitSTACK=32MB`) to bump the total ceiling to 8MB, and we sanitize environment variables before launch.
2. **`MAX_ARG_STRLEN`**: The maximum size of any *single* argument string.
   - *Default*: **128 KB** (131,072 bytes).
   - *The Catch*: This limit is a hardcoded kernel `#define` and **cannot** be raised via `ulimit` or `systemd`.

If the `--system-prompt` string exceeds 128 KB, the kernel will refuse to execute the subprocess, regardless of how much total `ARG_MAX` memory is available.

## Identifying the Bloat

During an operational incident, the prompt ballooned to **~132 KB**, immediately breaking session initialization. The bloat was caused by inefficiencies in dynamic prompt construction:

1. **YAML Frontmatter Dumps**: `agent_setup.py` was dumping the entire raw YAML frontmatter (including execution scripts and redundant descriptions) for every single installed skill into the `capabilities.md` block.
2. **Duplicate Injection**: `prompt_builder.py` appended a raw `<skills>` XML dump to the very end of the prompt, resulting in the skills list being printed twice.
3. **Visual Formatting Bloat**: The agent loop used heavy vertical formatting (`-> Delegate: ...`) repeated on a new line for every single micro-agent.
4. **Atlas VP Redundancy**: The VP worker prompt builder (`build_vp_system_prompt`) was similarly injecting the XML dump blindly.

## Architectural Guidelines for System Prompts

To solve the crash and prevent recurrence, we redesigned the prompt generation logic to optimize for density. The prompt dropped from 132 KB to ~75 KB. 

Going forward, any additions to the base System Prompt or Capabilities Registry must adhere to these rules:

### 1. High-Signal, High-Density Formatting
Lists of capabilities, skills, and agents must be formatted as compact, single-line Markdown bullets. Do not use multi-line descriptive paragraphs for index data. 

*Bad:*
```markdown
#### agent-browser
Automates browser interactions for web testing...
Source: /path/to/skill
```

*Good:*
```markdown
- **agent-browser** (/path/to/skill): Automates browser interactions for web testing...
```

### 2. Progressive Disclosure (No Deep Embeds)
Do not embed the *content* of external documents or long SOPs directly into the initial system prompt. 

Only provide an ultra-compact **index** (Name, Path, and a 1-sentence description). Instruct the agent to proactively use tools (like `view_file` or `mcp__internal__read_file`) to read the full context off the disk *only when exactly required*.

### 3. Shared Registry Prevention
Validate that dynamically loaded blocks (like `capabilities.md` generated by `agent_setup.py`) do not duplicate data (like `<skills>`) that another part of `prompt_builder.py` is also appending. Every duplicated kb pushes the system closer to the unchangeable 128KB execution cliff.

## Operational Follow-On: Sanitize the Child Env, Not the Parent Runtime

The 2026-03-23 tutorial-pipeline incident exposed a second architectural rule
that sits next to the prompt-size guidance above.

To avoid `E2BIG`, `src/universal_agent/execution_engine.py` sanitizes the
environment used to spawn the Claude SDK subprocess. That is correct. What is
**not** correct is letting that sanitization permanently mutate the long-running
gateway process environment.

Why this matters:

1. the child Claude subprocess does not need the full Infisical-loaded service
   environment
2. the parent gateway process still does need runtime secrets for unrelated
   features such as YouTube proxy ingest
3. if sanitization leaks into the parent process, later request handlers can
   behave as though secrets were never loaded

The correct pattern is now:

1. snapshot `os.environ`
2. sanitize only for the child spawn window
3. restore the original parent env immediately afterward

In the current implementation this is handled by
`_temporary_sanitized_process_env()` around
`SubprocessCLITransport.connect()`.

Architectural rule:

- use env sanitization as a narrowly scoped child-process compatibility tool
- never use it as a persistent parent-process cleanup step in a service runtime
