## 1. Overview

The **CSI Convergence Intelligence Pipeline** is a specialized subsystem designed to detect, synthesize, and report on emerging signals and macro-trends from the Creator Signal Intelligence (CSI) video feeds. It bridges the gap between raw, ingested transcripts and actionable intelligence surfaced in the Universal Agent Task Hub.

## 2. Dual-Track Detection Architecture

The pipeline employs a dual-track architecture to balance rapid programmatic detection with deeper LLM-driven synthesis:

### Track A: Concrete (Programmatic Fast-Pass)
- **Purpose**: Rapidly identify direct topical intersections between newly ingested video schemas and recent category history.
- **Mechanism**: Evaluates incoming signals against the `proactive_topic_signatures` table.
- **Gating**: Uses an LLM-based **Signal Strength** evaluator. Only signals scoring **>= 8** out of 10 are permitted to trigger tasks, minimizing dashboard noise.
- **Output**: Currently creates Task Hub items with IDs prefixed `convergence-brief:` and `source_kind='convergence_detection'`. The ToDo dashboard should map `convergence_detection` to the intended green pill.

### Track B: Abstract (LLM-Driven Synthesis)
- **Purpose**: Identify overarching macro-trends and abstract relationships across different domains over time.
- **Mechanism**: Batch processes collections of topic signatures to synthesize insights that algorithmic matching might miss.
- **Output**: Currently creates Task Hub items with IDs prefixed `insight-brief:` and `source_kind='insight_detection'`. The ToDo dashboard should map `insight_detection` to the intended amber pill.

## 3. Data Ingestion & Taxonomy

### The Ingestion Hook
Data can flow from the core CSI ingester to the convergence pipeline via `sync_topic_signatures_from_csi` in `src/universal_agent/services/proactive_convergence.py`. That helper reads transcript-backed rows from the CSI database, upserts `proactive_topic_signatures`, and may queue convergence or insight tasks.

Current wiring: `sync_topic_signatures_from_csi` is still called from `src/universal_agent/proactive_signals.py` during proactive signal sync, and it is also available through the cron runner `src/universal_agent/scripts/csi_convergence_sync.py`. The gateway registers a fixed `csi_convergence_sync` Chron job at startup when `UA_CSI_CONVERGENCE_CRON_ENABLED` is truthy. This gives CSI convergence a non-dashboard producer path while preserving the explicit dashboard refresh path.

Default producer configuration:

| Setting | Default | Purpose |
| --- | --- | --- |
| `UA_CSI_CONVERGENCE_CRON_ENABLED` | `1` | Register the runtime Chron job at gateway startup |
| `UA_CSI_CONVERGENCE_CRON_EXPR` | `*/30 * * * *` | Run every 30 minutes |
| `UA_CSI_CONVERGENCE_CRON_TIMEZONE` | `America/Chicago` | Chron timezone |
| `UA_CSI_CONVERGENCE_SYNC_LIMIT` | `400` | Max CSI analysis rows processed per sync |
| `CSI_DB_PATH` | `/var/lib/universal-agent/csi/csi.db` | Source CSI database |

## 5b. Hourly insight delivery (PR `claude/hourly-insight-redesign`, 2026-05-25)

Adds a structural lever that replaces Simone's per-brief email loop with a
single hourly digest. The convergence cron still generates briefs as before,
but delivery now flows through a deterministic scoring + selection pipeline:

```
csi_convergence_sync (cron, hourly)
  └── creates insight_brief_task / convergence_brief_task artifacts in
      activity_state.db with metadata.confidence (Track B) or
      metadata.signal_strength (Track A) and supporting_channel_count
hourly_insight_email (cron, top of every hour 6 AM-9 PM Houston)
  └── services/hourly_insight_email.compose_hourly_email scores every
      undelivered brief from the last hour via the composite:
        0.4*confidence
      + 0.3*min(supporting_channel_count/5, 1)
      + 0.2*(1 - max_jaccard_overlap_against_last_7d_briefs)
      + 0.1*normalized_preference_score
      Floor (≥3 channels AND confidence ≥0.7) is informational — sub-threshold
      briefs still fill the slot when no candidate clears.
  └── picks insight #1 (highest composite) and insight #2 (highest composite
      among candidates with Jaccard topic-overlap < 0.30 with #1).
  └── sends ONE `[Hourly Intel]` email from Simone's AgentMail inbox.
  └── stamps proactive_artifacts.delivered_at so the brief is excluded next
      hour, and writes a row per considered brief to
      proactive_brief_scoring_log for weekly tuning.
morning_briefing / evening_briefing
  └── recap section lists every insight that went out via hourly email since
      the prior briefing, then flips delivered_briefing=1 on those scoring-log
      rows so they aren't recapped twice.
  └── evening briefing only: picks ONE unsurfaced brief from the window as an
      "Honorable Mention" via LLM selection of the unsurfaced backlog.
insight_scoring_health (cron, Sunday 8 AM Houston)
  └── 7-day rollup: # generated, # delivered hourly, # delivered in briefing,
      sub-threshold rate, operator-rating distribution. LLM-generated verdict
      ("scoring is calibrated" / "raise confidence floor to 0.8" / …).
```

### Schema additions (activity_state.db)

- `proactive_artifacts.delivered_at TEXT` — stamped by the hourly cron once a brief is emailed so it isn't re-considered. Added via idempotent `ALTER TABLE ADD COLUMN` in `proactive_artifacts.ensure_schema`.
- `proactive_brief_scoring_log` — full table (see `proactive_scoring_log.ensure_schema`). Unique on `(artifact_id, delivery_slot, logged_at)` for safe re-runs.

### Prompt change (Track B ideation)

The Track-B `_IDEATION_SYSTEM` prompt now requires a `confidence` field on
each emitted insight (0.0-1.0 self-rating). This flows into
`metadata.confidence` on the resulting `insight_brief_task` artifact and is
the primary input to the composite score's 0.4× weighted term. Track A
already returned `signal_strength` (1-10) which we now persist as
`metadata.signal_strength` and normalize to [0, 1] for the same composite.

## 6. Future Optimizations

- **Signal Tuning**: The strict `Signal Strength >= 8` threshold may be tuned based on the observed frequency of generation.
- **Feedback Loops**: Future iterations aim to integrate implicit or explicit user feedback from the ToDo List directly back into the LLM's heuristic weightings.
- **Runtime Producer Metrics**: Surface sync counts, skipped-by-threshold counts, queued task counts, and LLM failure counts in Ops or the CSI dashboard.
- **Contract Tests**: Keep service, gateway endpoint, and ToDo source-kind tests aligned so the pipeline cannot silently drift.
