# Lossless Memory Subsystem

## Overview

The Lossless Memory subsystem (`src/universal_agent/lossless_memory/`) provides a pluggable, non-destructive, opt-in DAG-based (Directed Acyclic Graph) history condensation module for the Universal Agent.

Designed using the **Strangler Fig pattern**, it completely overrides the agent's context window management and summarization techniques when enabled, converting linear token-eviction strategies into a hierarchical, self-condensing memory graph that preserves the semantic richness of evicted context.

## Core Components

The subsystem consists of four main components isolated entirely off the root:

1. **`db.py` (LosslessDB Engine)**:
   The SQLite interface that maps out the DAG data structures. It creates and manages four tables:
    - `lcm_conversations`: Tracks the overarching sessions.
    - `lcm_messages`: Stores raw messages, tool calls, and structured JSON blocks exactly as passed by the SDK.
    - `lcm_summaries`: Stores the parent nodes generated by the condenser.
    - `lcm_context_items`: The ordered joint-table tracking the current linearized context window view.

2. **`history_adapter.py` (`LosslessMessageHistory`)**:
   A perfect duck-typed drop-in replacement for Universal Agent's native `MessageHistory`. It hooks into the main `add_message()` loop to serialize raw components directly to the database instead of counting placeholder tokens. It enforces truncation by calling the condenser when the `UA_TRUNCATION_THRESHOLD` is breached.

3. **`dag_condenser.py`**:
   The engine responsible for executing compaction sweeps. When triggered, it identifies evictable message nodes (older than the "Fresh Tail" count), runs them through an LLM to generate an XML-wrapped summary, creates a Depth-0 Summary Node, and updates the `lcm_context_items` view to replace the raw messages with their parent summary.

4. **`tools.py`** (Optional Extension):
   Drill-down tools (`lcm_grep`, `lcm_expand`) that can be exposed to the Universal Agent so it can fetch the underlying raw messages of a summarized block when deep context is required.

## Activation & Configuration

The entire subsystem is toggled via feature flags at the `agent_core.py` level.
Unlike the native memory system which evaluates pure strings, when this feature is active, actual tool content blocks must be passed to the history adapter to ensure the database correctly indexes JSON parameters and return values.

### Environment Variables

| Variable | Default | Description |
| --- | --- | --- |
| `UA_ENABLE_LOSSLESS_MEMORY` | `1` | Master feature flag. If `1`, the subsystem replaces native MessageHistory. |
| `UA_LOSSLESS_DB_PATH` | `~/.universal_agent/lcm.db` | Absolute path to the SQLite store. |
| `UA_LCM_FRESH_TAIL` | `10` | The number of recent messages immune to summarization sweeps. |
| `UA_TRUNCATION_THRESHOLD` | `150000` | Context saturation limit before compaction triggers. |
| `UA_CONTEXT_WINDOW` | `200000` | Absolute hard wall limit for the LLM context. |

## Recovery & Rollback

Because of the non-destructive parallel design, rolling back is instantaneous and mathematically safe.

If the agent experiences "Summarization Loops" (a known bug in DAG memory where the condenser becomes trapped re-summarizing its own summaries) or general instability:
1. Set `UA_ENABLE_LOSSLESS_MEMORY=0` or remove it entirely.
2. Restart the runtime.
3. The Universal Agent immediately reverts back to standard memory truncation behavior with `MessageHistory` and zero database dependencies.
